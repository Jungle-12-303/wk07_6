# 메모리 계층 구조와 접근 비용

CPU는 한 사이클에 모든 메모리에 접근할 수 없다. 더 빠른 메모리는 더 비싸고 더 작고, 더 큰 메모리는 더 멀고 더 느리다. 이 물리적 제약이 오늘날 우리가 쓰는 모든 컴퓨터를 **메모리 계층(Memory Hierarchy)** 구조로 만든다.

## 왜 계층이 필요한가

전자가 도체 안에서 움직일 수 있는 속도에는 상한이 있다. CPU 클럭이 3 GHz라면 한 사이클은 약 0.33 ns이고, 그 동안 빛이 진공에서 갈 수 있는 거리는 약 10 cm다. 데이터는 CPU 코어 내부에서 나와 메인 메모리 모듈까지 왕복해야 하는데, 거리만으로도 수십 사이클이 사라진다. 여기에 DRAM 셀을 재충전하는 시간, 신호를 정렬하는 시간이 더해져 **DRAM 접근 한 번은 수백 사이클**이 든다.

그렇다고 메모리 전부를 SRAM으로 만들 수는 없다. SRAM 한 비트는 트랜지스터 6개로 구성되는 반면 DRAM 한 비트는 트랜지스터 1개와 캐패시터 1개로 구성된다. 면적·전력·원가 모두 SRAM이 한 자릿수 이상 비싸다. 결국 빠른 메모리는 소량만 CPU 옆에 두고, 느린 메모리는 대용량을 멀리 두는 타협이 선택된다. 이 타협이 계층이다.

## 계층의 모습

```
         ┌─────────────────────────────┐
         │  Registers   ~ 수십 바이트   │  0 cycle
         ├─────────────────────────────┤
         │  L1 Cache    ~ 32 KB         │  ~4 cycle
         ├─────────────────────────────┤
         │  L2 Cache    ~ 256 KB ~ 1 MB │  ~12 cycle
         ├─────────────────────────────┤
         │  L3 Cache    ~ 수 MB ~ 수십   │  ~40 cycle
         ├─────────────────────────────┤
         │  DRAM        ~ 수 GB ~ 수백   │  ~200 cycle
         ├─────────────────────────────┤
         │  SSD/NVMe    ~ 수백 GB ~ 수 TB│  ~수십 μs
         ├─────────────────────────────┤
         │  HDD         ~ 수 TB          │  ~수 ms
         └─────────────────────────────┘
           위로 갈수록  빠르고 작고 비싸다
           아래로 갈수록 느리고 크고 싸다
```

```mermaid
flowchart LR
    CPU["CPU 코어"] -->|0 cycle| R["레지스터"]
    R -->|~4 cycle| L1["L1 캐시"]
    L1 -->|~12 cycle| L2["L2 캐시"]
    L2 -->|~40 cycle| L3["L3 캐시"]
    L3 -->|~200 cycle| DRAM["DRAM"]
    DRAM -->|~수십 μs| SSD["SSD/NVMe"]
    SSD -->|~수 ms| HDD["HDD"]
```

각 층은 바로 아래 층의 **캐시** 역할을 한다. L1은 L2의 캐시, L2는 L3의 캐시, L3는 DRAM의 캐시, DRAM은 디스크의 캐시다. 이 관계는 재귀적이다. 한 층에서 찾지 못하면 다음 층으로 내려간다.

## 각 층의 정체

**레지스터(Register)** 는 CPU 코어 안에 있는 저장 공간이다. x86-64의 범용 레지스터는 16개이며 각각 64비트, 즉 코어당 128바이트 정도다. 명령어가 직접 이름으로 지목한다. 로드·저장 명령이 아니라 **즉시 사용**된다.

**L1 캐시(L1 Cache)** 는 코어마다 분리돼 있고, 보통 명령어 캐시(L1i)와 데이터 캐시(L1d)가 따로 있다. 크기는 32~48 KB, 히트 지연은 약 4 사이클이다. 캐시 라인(Cache Line) 단위(대개 64바이트)로 데이터를 보관한다.

**L2 캐시(L2 Cache)** 도 코어별 개인 캐시지만 L1보다 크고 느리다. L1에서 miss가 난 라인을 L2에서 찾는다.

**L3 캐시(L3 Cache, LLC)** 는 소켓 안의 모든 코어가 공유하는 마지막 방어선이다. 멀티코어 캐시 일관성 프로토콜(MESI 등)이 이 층에서 조율된다.

**DRAM** 은 메인보드의 DIMM 슬롯에 꽂히는 모듈이다. 용량은 GB 단위로 크지만, 캐패시터 기반이라 주기적 재충전(refresh)이 필요하고 접근 지연이 ns 단위다.

**SSD/HDD** 는 전원이 꺼져도 데이터가 유지되는 영속 저장소다. DRAM보다 수만~수십만 배 느리다. 그래서 운영체제는 디스크의 내용을 DRAM에 캐싱한다. 이것이 **페이지 캐시(Page Cache)** 라고 불리는 OS의 핵심 메커니즘이다.

## 단위: 계층마다 다르다

한 번에 이동하는 데이터의 단위가 층마다 다르다.

| 층       | 이동 단위                 | 관찰 가능한 단위 |
| -------- | ------------------------- | --------------- |
| 레지스터 | 1~8 바이트                | 바이트          |
| L1~L3    | **캐시 라인** (보통 64 B) | 64 바이트       |
| DRAM     | 캐시 라인                 | 64 바이트       |
| 디스크   | **페이지** (보통 4 KB)    | 4 KB 블록       |

이 단위의 차이가 성능을 지배한다. "한 바이트만 읽어도 실제로는 64바이트가 움직인다"거나 "한 바이트만 고쳐도 실제로는 4 KB가 디스크에서 읽히고 다시 4 KB가 기록된다"는 사실은 코드를 작성하는 순간엔 보이지 않지만, 물리적으로는 매 접근마다 일어난다.

## 프로그래밍에 미치는 영향

계층이 존재하는 한, 성능을 결정하는 것은 **접근 패턴**이다.

```mermaid
flowchart TD
    A["CPU가 주소 X를 요청"] --> B{"L1 hit?"}
    B -- "예" --> Z["4 cycle에 완료"]
    B -- "아니오" --> C{"L2 hit?"}
    C -- "예" --> Y["~12 cycle"]
    C -- "아니오" --> D{"L3 hit?"}
    D -- "예" --> X["~40 cycle"]
    D -- "아니오" --> E{"DRAM hit?"}
    E -- "예" --> W["~200 cycle"]
    E -- "아니오" --> F["디스크에서 페이지 로드<br/>~수 ms"]
```

한 번의 L1 hit과 한 번의 페이지 폴트는 대략 **10^6 배**의 성능 차이가 난다. 그래서 다음 두 원리가 성립한다.

- **시간적 지역성(Temporal Locality)**: 방금 쓴 데이터는 곧 다시 쓰인다. 쓴 직후에 캐시에 살아 있다면, 재접근이 싸다.
- **공간적 지역성(Spatial Locality)**: 쓴 데이터 옆에 있는 데이터도 곧 쓰인다. 한 캐시 라인이 올라올 때 주변 바이트들이 같이 올라오므로, 인접 주소를 연속으로 접근하면 "공짜로" 따라온 데이터를 활용할 수 있다.

두 지역성을 가지도록 자료구조를 배치하고 코드를 배열하는 일이 곧 캐시 친화적인 프로그래밍이다. 연결 리스트 대신 배열, 이중 포인터 대신 구조체 배열(AoS/SoA 선택), 랜덤 접근 대신 순차 접근이 일관되게 빠른 이유가 이 계층 때문이다.

## 정리

메모리 계층은 선택이 아니라 물리의 결과다. 작고 빠른 것과 크고 느린 것은 함께 존재할 수 없으므로, 시스템은 작고 빠른 층이 크고 느린 층의 캐시가 되는 구조로 타협한다. 프로그래머가 쓰는 "메모리 접근 한 번"은 실제로는 6~7층짜리 계층을 위에서부터 차례로 두드리는 행위이며, 그 중 어느 층에서 "있다"는 응답이 오느냐가 성능을 결정한다. 그리고 그 응답을 유리하게 만드는 유일한 레버는 **접근 패턴을 지역성에 맞추는 것** 뿐이다.
