# 01. 물리 세계 — CPU, 캐시, DRAM이 실제로 움직이는 방식

> 키워드 트리에서 이 글의 위치: `Memory Systems → Program → C → compiler → machine code → load/store → CPU`
>
> 이 글은 가장 아래층에서 시작한다. `int x = arr[i];` 한 줄이 하드웨어에서 어떤 물리적 동작을 만드는지를 따라가 본다.

## 들어가며 — 이번 주에 왜 여기서부터 시작했나

이번 주에 CSAPP 9장을 잡고 가장 먼저 충격을 받은 지점은, 내가 평소에 "메모리" 라고 부르던 것이 사실은
CPU · 캐시 · DRAM · 디스크 라는 **서로 다른 속도의 저장 계층**이 층층이 쌓여 있는 구조라는 사실이었다.

minidb(디스크 기반 SQL 엔진)를 직접 쌓아 올리기 전까지 나에게 메모리는 그냥 "변수를 담아 두는 곳"이었다.
그런데 B+ tree 의 노드 크기를 `sysconf(_SC_PAGESIZE)` 로 잡고, 4096바이트 단위로 페이지를 관리하기 시작하자
내 코드 한 줄 한 줄이 **어느 물리적 저장소**에 접근하는지 의식하게 됐다.
그래서 이 시리즈는 가장 물리적인 층에서 시작한다.

## CPU는 오직 load / store 로만 메모리를 만진다

C에서 `int x = arr[i];` 는 여러 개념(변수, 배열, 인덱스)이 섞여 있지만,
컴파일러가 뱉어내는 기계어는 결국 두 가지 명령 중 하나로 수렴한다.

- `load` — 어떤 주소에 있는 값을 레지스터로 가져오기
- `store` — 레지스터의 값을 어떤 주소에 쓰기

즉 C의 변수, 포인터, struct, array 는 모두 **"주소 계산을 어떻게 할 것인가"** 에 대한 문법이고,
실제 실행 단계에선 전부 `load / store` 한 쌍으로 풀려버린다.

```
C 코드:
    int x = arr[i];

하드웨어 관점:
    1) 레지스터에 base(arr)와 index(i)를 올린다.
    2) base + i * sizeof(int) 로 주소 계산
    3) 그 주소에서 4바이트 load → 레지스터
    4) 레지스터 값을 x 의 주소에 store
```

이 지점에서 나는 CSAPP 9장의 키워드 트리를 다시 읽게 됐다.
트리의 아주 윗단은 `source code, object, pointer` 같은 **"C 문법 레벨"** 의 단어였는데,
그 아래로 내려가면 전부 `load / store → CPU → register` 로 수렴했다.

C 코드 → 컴파일러 → 기계어 → CPU 라는 흐름을 한 번 지나고 나면,
**"C에서 포인터를 안다"는 말이 사실은 "내가 어떤 주소를 계산하고 있는지를 안다"는 뜻**이라는 게 명확해진다.

## CPU 안쪽 — 레지스터는 "번지가 없는 초고속 메모리"

CPU 안에 있는 저장소는 세 종류로 나뉜다.

| 저장소 | 크기 | 접근 비용 | 식별 방식 |
|--------|------|-----------|-----------|
| register | 수십 개 × word | ~1 cycle | 이름 (RAX, RSP, PC …) |
| cache (L1/L2/L3) | KB~MB | ~1~40 cycle | 주소 (tag/index/offset) |
| DRAM | GB | ~100 cycle+ | 주소 (row/col) |

레지스터는 **주소가 없다**. `mov rax, rbx` 는 "RBX 레지스터의 값을 RAX 레지스터로 옮겨라" 지,
"주소 0x1234의 값을 주소 0x5678로 옮겨라" 가 아니다.
즉 `load / store` 는 **레지스터와 메모리 주소 사이**에서만 일어난다.

특히 두 레지스터는 메모리 시스템 이해의 핵심 축이다.

- `PC` (Program Counter) — 다음에 실행할 명령의 **주소**
- `SP` (Stack Pointer) — 현재 스택의 top **주소**

이 둘만 봐도 CPU가 코드든 데이터든 전부 "주소"로 추적한다는 사실이 드러난다.
그리고 이 주소는 아직 가상 주소(VA) 일 뿐, 곧이어 MMU 가 물리 주소(PA)로 번역해 줄 것이다 (다음 글).

## memory access granularity — CPU가 "한 번에" 옮기는 단위

CPU는 바이트 단위로 메모리를 다룰 수 있지만, 실제로 **물리 계층이 한 번에 옮기는 단위**는 다르다.
키워드 트리에서 이 부분을 처음 봤을 땐 네 개가 다 헷갈렸는데, 정리하면 다음과 같다.

| 단위 | 크기 | 누가 다루는 단위인가 |
|------|------|----------------------|
| byte | 1B | 프로그래머가 주소로 가리키는 최소 단위 |
| word | 8B (x86-64) | CPU가 자연스럽게 다루는 데이터 폭 |
| cache line | 64B | **캐시가 DRAM에서 한 번에 가져오는 단위** |
| page | 4KB | **OS / MMU 가 매핑·보호·I/O 를 다루는 단위** |

여기서 중요한 두 가지.

**1) CPU가 `load byte` 를 하더라도 DRAM에서는 64바이트가 한 번에 올라온다.**
cache line 전체가 L1 으로 이동한다. 바이트 하나를 읽고 끝나는 게 아니다.

**2) OS와 MMU는 4KB page 단위로만 메모리를 관리한다.**
`mmap`, `brk`, page table entry, page fault … 모두 4KB 단위 사건이다.
그래서 64B 바뀐다고 page 전체가 dirty 처리된다.

이 감각이 minidb 구현에 직접적으로 작용했다.
**minidb는 모든 디스크 I/O 단위를 4096바이트 page 로 맞췄고, 한 page 에 한 B+ tree 노드를 올렸다.**
노드가 작아지면 디스크가 한 번에 가져오는 4KB 안에서 공간이 낭비되고,
노드가 커지면 B+ tree 의 fan-out 이 올라가 높이가 극단적으로 낮아진다. (→ 포스트 09에서 더 구체)

## cache hierarchy — 속도 계층

DRAM 접근은 ~100 cycle, L1 cache 는 ~4 cycle. 25배 차이다.
그래서 CPU는 가장 최근에 쓴, 그리고 공간적으로 가까운 데이터를 자동으로 L1/L2/L3 에 올려 둔다.

이 계층이 의미를 가지려면 **locality** 라는 전제가 필요하다.

- `temporal locality` — 최근에 접근한 걸 또 접근한다 (반복문의 같은 변수)
- `spatial locality` — 가까운 주소를 연속으로 접근한다 (배열 순회)

둘 다 "프로그래머가 쓰는 자연스러운 패턴" 이지만,
struct 를 어떻게 배치하느냐, 배열을 어느 방향으로 순회하느냐에 따라 cache hit rate 는 크게 달라진다.

```c
// 좋음 — 연속된 cache line 을 순회
for (int i = 0; i < N; i++) sum += arr[i];

// 나쁨 — 매 iteration 마다 다른 cache line
for (int i = 0; i < N; i++) sum += arr[i * STRIDE];
```

그리고 여기서 한 단계 더 들어가면 **false sharing** 이 등장한다.
두 스레드가 "서로 다른 변수" 를 건드려도, 그 두 변수가 같은 64B cache line 안에 들어 있으면
cache coherence 프로토콜이 계속 같은 line 을 주고받느라 성능이 무너진다.
CSAPP 에서도, 그리고 이번 주 팀 리뷰에서도 이 주제가 여러 번 등장했다.

## minidb 에 적용한 방식 — "cache line 을 침해하지 말 것"

minidb 에서 같은 원리를 의식적으로 적용한 지점이 두 군데 있다.

**1) `frame_t` 가 아주 자주 접근되는 `used_tick`, `pin_count`, `is_dirty` 를 같은 struct 에 모아 뒀다.**
LRU 루프는 모든 frame 의 이 세 필드를 순회하면서 "가장 오래된 unpinned" 프레임을 찾아야 한다.
struct 가 작을수록 한 cache line 안에 더 많은 frame 메타가 들어가 순회가 빨라진다.

**2) page 헤더의 필드 순서를 "자주 같이 읽히는 것끼리" 묶었다.**
예를 들어 leaf page 헤더에서 `key_count` 와 `next_leaf_page_id` 는 레인지 스캔에서 같이 쓰이므로
헤더의 앞부분에 연속으로 배치했다.

세부 코드보다 더 중요한 건, 이 결정을 할 때 내가 "이건 cache line 의 문제다" 라는 말을
실제로 할 수 있게 됐다는 점이었다. 이번 주 학습에서 가장 뚜렷한 변화 하나다.

## DRAM 과 그 너머 — NUMA, DMA

cache 아래는 DRAM 이고, DRAM 은 메모리 컨트롤러가 row/bank 타이밍에 맞춰서 읽는다.
멀티 소켓 시스템에서는 core 와 DRAM 의 거리가 소켓마다 달라지기 때문에 같은 주소라도 접근 비용이 다르다 (`NUMA`).
또 디바이스가 CPU 를 거치지 않고 DRAM 에 직접 쓰는 `DMA` 경로도 있는데,
`IOMMU` 가 이 접근을 번역·보호해 주지 않으면 보안 문제가 된다.

minidb 는 NUMA / DMA 경계를 직접 만지지는 않았다. 하지만 "디스크 → 페이지 캐시 → 프레임 캐시 → CPU 레지스터" 로 이어지는
긴 경로 전체가, 결국 이 물리 세계의 여러 저장소 사이를 값이 오가는 과정이라는 것을 실감했다.

## 이번 주 인사이트 — "물리 세계"가 코드 한 줄에 개입한다

이번 주 가장 인상 깊었던 장면은, B+ tree 의 노드 크기를 정하는 자리에서
"교과서에는 fan-out 을 ORDER 라고 불러요" 라는 설명이 아니라
"4096 안에서 몇 개의 엔트리가 맞을지를 역산하자" 라는 **물리 단위 기반 설계** 를 하게 된 점이다.

- 프로그래머 언어에서는 노드가 `struct Node { Key keys[ORDER]; };` 로 시작한다.
- CSAPP 9장의 언어에서는 노드가 **cache line 과 page 라는 물리 단위** 위에서 정의된다.

이번 주 이전에는 이 둘이 완전히 다른 층에 있었다. 이번 주를 지나고 나서는
"알고리즘의 의도" 와 "하드웨어의 선호" 가 같은 설계 결정 안에서 만나는 지점을 직접 만들어 봤다는 감각이 생겼다.
