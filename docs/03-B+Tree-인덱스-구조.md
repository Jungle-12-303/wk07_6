# B+ Tree 인덱스 구조 — 할당, 정렬, 분할, 승격의 전체 흐름

## 1. 왜 페이지를 노드로 사용하는가?

### 교과서의 B+ tree vs minidb의 B+ tree

교과서에서 B+ tree는 보통 이렇게 표현된다:

```
교과서:
  노드 = 메모리의 객체 (malloc된 구조체)
  자식 연결 = 포인터 (메모리 주소)
  트리 전체가 메모리에 상주

  struct node {
      int keys[ORDER];
      struct node *children[ORDER+1];  ← 메모리 포인터
  };
```

하지만 데이터베이스는 **디스크 기반**이다. 메모리 포인터는 프로그램이 종료되면 사라진다.

```
minidb:
  노드 = 디스크의 페이지 (4096바이트 블록)
  자식 연결 = page_id (파일 내 위치를 나타내는 정수)
  필요한 노드만 디스크에서 읽어옴

  페이지 안에:
  [page_type][parent_page_id][key_count][leftmost_child_page_id]
  [key=146, right_child_page_id=4]
  [key=292, right_child_page_id=6]
  ...
```

### 페이지 = 노드 설계의 이점

**1) 디스크 I/O 최소화**

디스크에서 데이터를 읽을 때 최소 단위는 OS 페이지(4096바이트)이다.
노드 하나가 100바이트여도 디스크에서는 4096바이트를 읽어야 한다.
그렇다면 4096바이트를 통째로 하나의 노드로 쓰는 게 효율적이다.

```
노드가 작을 때 (100바이트 노드):
  1회 디스크 읽기 = 4096바이트 → 노드 1개만 사용 → 3996바이트 낭비
  트리 높이: 깊어짐 → 더 많은 디스크 읽기 필요

페이지 = 노드 (4096바이트 노드):
  1회 디스크 읽기 = 4096바이트 → 노드 1개 전체 활용 → 낭비 없음
  내부 노드 하나에 key 340개 → 자식 341개를 안내
  → 트리가 매우 납작해짐 → 디스크 읽기 횟수 최소화
```

**2) 트리 높이가 극단적으로 낮아진다**

```
이진 트리(key 2개/노드)로 1000만 건 검색:
  높이 = log₂(10,000,000) ≈ 24단
  → 24번 디스크 읽기

B+ tree(key 340개/노드)로 1000만 건 검색:
  높이 = log₃₄₁(10,000,000) ≈ 3단
  → 3번 디스크 읽기

디스크 접근 8배 감소
```

**3) OS 페이지 캐시와 정렬**

OS 커널도 파일 I/O를 페이지 단위로 캐싱한다. DB 페이지 크기가 OS 페이지 크기와 일치하면
하나의 DB 페이지가 정확히 하나의 OS 캐시 라인에 대응하여 효율적이다.

```
sysconf(_SC_PAGESIZE):
  x86_64:       4096바이트 = DB 페이지 1개
  Apple Silicon: 16384바이트 = DB 페이지 1개

→ DB 페이지 읽기 = OS 캐시 라인 1개 로드 = 낭비 없는 I/O
```

**4) 관리 단순화 — 별도 구조체 불필요**

```
C++/Java라면:
  class BTreeNode {
      vector<int> keys;
      vector<BTreeNode*> children;
      BTreeNode* parent;
  };
  → 별도 메모리 관리, 직렬화/역직렬화 필요

C + 페이지 방식:
  페이지 자체가 노드. memcpy로 즉시 디스크에 쓰고, 읽으면 바로 사용.
  별도 직렬화 불필요. packed 구조체가 곧 디스크 포맷.
```

## 2. 초기 상태 — 빈 트리

`pager_open(create=true)` 실행 직후:

```
page 2 (리프, 루트):
  ┌─────────────────────────────────────────────┐
  │ type=LEAF, parent=0, key_count=0            │
  │ next_leaf=0, prev_leaf=0                    │
  │                                             │
  │ (엔트리 없음 — 빈 리프)                      │
  └─────────────────────────────────────────────┘

헤더: root_index_page_id = 2
트리 높이: 1 (리프만 있음)
```

## 3. INSERT 과정 — 리프에 공간이 있을 때

```
INSERT id=1 → INSERT id=2 → INSERT id=3

page 2 (리프, 루트):
  ┌────────────────────────────────────────────────────────────┐
  │ type=LEAF, parent=0, key_count=3                          │
  │ [key=1, ref={p1,s0}] [key=2, ref={p1,s1}] [key=3, ref={p1,s2}] │
  └────────────────────────────────────────────────────────────┘
```

정렬 삽입 과정 (key=2 삽입 시, 기존 key=1, key=3):

```
기존:  [key=1] [key=3]
삽입:  key=2, 위치=1 (이진 탐색으로 결정)

임시 배열(tmp)에 조립:
  ① memcpy: tmp[0] = entries[0]        → [key=1]
  ② 직접:   tmp[1] = {key=2, ref}      → [key=1][key=2]
  ③ memcpy: tmp[2] = entries[1]        → [key=1][key=2][key=3]

tmp를 페이지에 다시 기록
```

## 4. 리프 분할 — 291개 초과 시

### 분할 과정 상세

```
page 2 (리프, 291개 꽉 참):
  [key=1] [key=2] [key=3] ... [key=145] [key=146] ... [key=291]

key=292 삽입 시도 → 공간 없음 → 분할 발생
```

**STEP 1: 임시 배열에 292개 조립**

```
tmp[292]:
  [key=1] [key=2] ... [key=145] | [key=146] [key=147] ... [key=291] [key=292]
                                ↑
                           split = 146 (절반 지점)
```

**STEP 2: 앞쪽 절반은 기존 페이지에**

```
page 2 (리프):
  key_count = 145
  [key=1] [key=2] ... [key=145]
```

**STEP 3: 뒤쪽 절반은 새 페이지에**

```
page 4 (새 리프, pager_alloc_page()로 할당):
  key_count = 147
  [key=146] [key=147] ... [key=292]
```

**STEP 4: 형제 연결 갱신**

```
page 2: next_leaf = page 4
page 4: prev_leaf = page 2, next_leaf = 0
```

**STEP 5: 승격 — 부모에 구분 key 전달**

```
promote_key = 146 (뒤쪽 페이지의 첫 번째 key)
→ "146 이상은 page 4로 가라"는 정보를 부모에 삽입
```

## 5. 새 루트 생성 — 최초 분할 시

page 2가 루트였으므로 부모가 없다. 새 루트를 만든다.

```
분할 전:
  root = page 2 (리프)

분할 후:
  page 5 (새 내부 노드, 새 루트):
    ┌─────────────────────────────────────────────────┐
    │ type=INTERNAL, parent=0, key_count=1            │
    │ leftmost_child = page 2                         │
    │ [key=146, right_child=page 4]                   │
    └─────────────────────────────────────────────────┘
    "146 미만 → page 2, 146 이상 → page 4"

  page 2 (리프): parent = page 5, keys 1~145
  page 4 (리프): parent = page 5, keys 146~292

헤더: root_index_page_id = 2 → 5로 변경
트리 높이: 1 → 2
```

### 트리 시각화

```
       [page 5: 내부 노드]
        key=146
       /        \
[page 2: 리프]  [page 4: 리프]
 key 1~145       key 146~292
```

## 6. 연쇄 분할 — 계속 INSERT 시

```
INSERT가 계속되어 리프 4개로 성장:

              [page 5: 내부 노드]
         key=146, key=292, key=438
        /       |        |        \
  [page 2]  [page 4]  [page 6]  [page 8]
  key 1~145  key 146~291 key 292~437 key 438~584
```

page 5의 내부 노드도 최대 340개 key까지 저장 가능.
341번째 key가 승격되면 page 5도 분할:

```
STEP 1: page 5 분할
  page 5:  앞쪽 170개 key
  page 10: 뒤쪽 169개 key (새 페이지)
  promote_key = 중간값

STEP 2: 새 루트 생성
  page 11 (새 루트):
    leftmost_child = page 5
    [key=중간값, right_child=page 10]

헤더: root_index_page_id = 5 → 11로 변경
트리 높이: 2 → 3
```

```
             [page 11: 루트]
              key=중간값
             /            \
    [page 5: 내부]     [page 10: 내부]
     key 170개           key 169개
    / | | ... |         / | | ... |
  [리프들]            [리프들]
```

## 7. 삭제 과정

### 리프에서 엔트리 제거

```
DELETE key=300 (page 6 리프에 있다고 가정):

page 6 리프:
  삭제 전: [key=292] [key=293] ... [key=300] [key=301] ... [key=437]
                                    ↑ 이걸 제거

  ① key=300의 위치(idx) 찾기 (이진 탐색)
  ② idx 뒤의 엔트리를 한 칸씩 앞으로 당김 (memcpy)
  ③ key_count-- (146 → 145)

  삭제 후: [key=292] [key=293] ... [key=299] [key=301] ... [key=437]
```

minidb는 단순 구현으로, 리프가 비어도 노드 병합(merge)은 하지 않는다.
리프의 모든 key가 삭제되면 해당 페이지를 free 리스트에 반환한다.

## 9. B+ tree의 핵심 불변식 (Invariant)

```
1. 모든 데이터(row_ref)는 리프에만 존재한다 (내부 노드에는 없다)
2. 리프는 key 순서로 정렬되어 있다
3. 리프끼리 이중 연결 리스트로 연결되어 있다
4. 내부 노드의 key는 자식을 분류하는 "구분자"일 뿐이다
5. 루트를 제외한 모든 노드는 최소 절반 이상 채워져 있다
6. 트리는 항상 균형 (모든 리프의 깊이가 동일)
7. 트리는 아래에서 위로 자란다 (분할 → 승격)
```
