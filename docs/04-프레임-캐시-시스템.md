# 프레임 캐시 시스템 — MMU/TLB처럼 페이지를 캐싱하는 방식

## 1. 왜 캐시가 필요한가

디스크 읽기(pread)는 메모리 접근보다 **수천~수만 배 느리다**.

```
메모리 접근:   ~100ns
SSD 읽기:     ~100,000ns (100μs)
HDD 읽기:     ~10,000,000ns (10ms)
```

매번 디스크에서 읽으면 `SELECT WHERE id=500`이 페이지 4번 × 100μs = 400μs 걸리지만,
캐시에 있으면 페이지 4번 × 100ns = 400ns로 **1000배 빨라진다**.

---

## 2. 프레임 — 메모리에 캐시된 페이지

프레임은 "디스크 페이지의 메모리 복사본"이다. minidb는 256개의 프레임을 유지한다.

### 프레임 구조 (frame_t)

```
typedef struct {
    uint32_t page_id;    // 이 프레임에 적재된 페이지 번호
    bool     is_valid;   // 유효한 데이터가 있는가
    bool     is_dirty;   // 수정되어 디스크 기록이 필요한가
    uint32_t pin_count;  // 현재 사용 중인 참조 수
    uint64_t used_tick;  // 마지막 접근 시점 (LRU 판별용)
    uint8_t *data;       // 페이지 데이터 버퍼 (4096바이트)
} frame_t;
```

### MMU/TLB와의 비교

```
┌───────────────────────────────────────────────────────────────────┐
│              MMU/TLB (OS)              │   pager 프레임 캐시       │
├────────────────────────────────────────┼──────────────────────────┤
│ 가상 주소 → 물리 주소 변환              │ page_id → frame 인덱스    │
│ TLB에 캐시됨 → 바로 접근               │ frame에 있음 → 바로 반환  │
│ TLB 미스 → 페이지 테이블 워크           │ frame 미스 → 디스크 pread │
│ 페이지 부재 → 디스크에서 로드           │ 빈 프레임 없음 → LRU 교체 │
│ dirty 페이지 → swap out 시 디스크 기록  │ dirty 프레임 → flush 시 pwrite │
│ 참조 비트 / accessed 비트              │ used_tick (LRU 시계)      │
│ 고정(pinned) 페이지 → 교체 불가        │ pin_count > 0 → 교체 불가 │
└────────────────────────────────────────┴──────────────────────────┘
```

## 4. 캐시 히트 vs 캐시 미스

### 캐시 히트 — O(256) 선형 탐색으로 프레임 발견

```
pager_get_page(page_id=5):

  frames[0]: page_id=3, valid ← 아님
  frames[1]: page_id=7, valid ← 아님
  frames[2]: page_id=5, valid ← 찾았다!

  → pin_count++  (1 → 2, 사용 중 표시)
  → used_tick = ++global_tick  (LRU 시계 갱신)
  → frames[2].data 포인터 반환

  디스크 접근: 0회
```

### 캐시 미스 — LRU 교체 후 디스크에서 읽기

```
pager_get_page(page_id=100):

  frames[0..255] 전체 탐색 → page_id=100 없음

  LRU 교체:
    1단계: 빈 프레임(is_valid=false) 있으면 바로 사용
    2단계: 없으면 pin_count=0인 프레임 중 used_tick이 가장 작은 것 선택
    3단계: 선택된 프레임이 dirty면 디스크에 먼저 기록

  frames[47]이 선택됨 (pin=0, tick=5, dirty):
    ① pwrite(fd, frames[47].data, 4096, 47번페이지오프셋)  ← dirty 기록
    ② pread(fd, frames[47].data, 4096, 100*4096)           ← 새 페이지 로드
    ③ frames[47].page_id = 100
    ④ frames[47].is_valid = true
    ⑤ frames[47].is_dirty = false
    ⑥ frames[47].pin_count = 1
    ⑦ frames[47].used_tick = ++global_tick

  디스크 접근: 최대 2회 (dirty flush + 새 페이지 read)
```

## 5. Pin/Unpin — 사용 중 보호

pin_count는 "이 프레임을 현재 몇 군데에서 사용 중인가"를 나타낸다.
pin_count > 0이면 **절대 교체되지 않는다.**

```
사용 패턴:

  uint8_t *page = pager_get_page(pager, 5);  // pin_count: 0 → 1
  // ... 페이지 내용 읽기/수정 ...
  pager_unpin(pager, 5);                      // pin_count: 1 → 0
                                               // 이제 LRU 교체 대상

위험한 상황 — pin을 풀지 않으면:
  pager_get_page(pager, 5);   // pin=1
  pager_get_page(pager, 5);   // pin=2 (같은 페이지 중첩 접근)
  pager_unpin(pager, 5);      // pin=1
  // unpin을 한 번만 하면 pin=1로 남아서 영원히 교체 불가
  // → 프레임 누수 → 결국 모든 프레임이 pin → 캐시 고갈
```

### 실제 호출 흐름 예시

```
bptree_search(key=500):
  ┌─ page 5 (루트) get_page → pin=1
  │   내부 노드 읽기 → "page 4로 가라"
  └─ page 5 unpin → pin=0

  ┌─ page 4 (리프) get_page → pin=1
  │   이진 탐색 → ref={page7, slot10}
  └─ page 4 unpin → pin=0

  ┌─ page 7 (힙) get_page → pin=1
  │   행 데이터 읽기
  │   ※ 호출자(executor)가 unpin해야 함
  └─ (executor에서) page 7 unpin → pin=0
```

## 6. LRU 교체 알고리즘

### used_tick — 전역 시계

```
모든 pager_get_page() 호출 시 global tick이 1씩 증가한다.
접근된 프레임의 used_tick이 현재 tick으로 갱신된다.

시간 순서:
  tick=1: page 5 접근 → frames[0].used_tick = 1
  tick=2: page 3 접근 → frames[1].used_tick = 2
  tick=3: page 5 접근 → frames[0].used_tick = 3  ← 갱신됨
  tick=4: page 7 접근 → frames[2].used_tick = 4

교체 시: pin_count=0인 프레임 중 used_tick이 가장 작은 것 선택
  → frames[1] (tick=2, page 3) 이 교체 대상
  → "가장 오래 전에 접근된 페이지"를 교체
```

### 교체 우선순위

```
1순위: 빈 프레임 (is_valid=false) → 바로 사용, 교체 비용 0
2순위: clean + unpinned (is_dirty=false, pin=0) → 그냥 덮어씀
3순위: dirty + unpinned (is_dirty=true, pin=0) → 디스크에 먼저 기록 후 덮어씀
불가:  pinned (pin_count>0) → 사용 중이므로 교체 불가
```

## 7. Dirty 프레임과 워터마크 자동 플러시

### dirty 표시

```
페이지 내용을 수정한 후 pager_mark_dirty(page_id) 호출:
  → 해당 프레임의 is_dirty = true
  → flush 전까지 메모리에만 존재, 디스크에는 반영 안 됨
  → 프로그램 비정상 종료 시 dirty 데이터 유실 가능
```

### 워터마크 기반 선제 플러시

dirty 프레임이 너무 많으면 한꺼번에 flush해야 하는 부담이 커진다.
이를 방지하기 위해 **워터마크(watermark)** 기반 자동 플러시를 수행한다.

```
dirty_high_watermark = 64  (이 이상이면 플러시 시작)
dirty_low_watermark  = 16  (이 수까지 줄임)

pager_mark_dirty() 호출 시:
  dirty 프레임 수 계산
  if (dirty >= 64):
    LRU 순서로 dirty+unpinned 프레임을 디스크에 기록
    dirty가 16개 이하가 될 때까지 반복

효과:
  INSERT 100회 → dirty 프레임 ~64개 쌓임 → 48개 자동 플러시
  → 프로그램 종료 시 flush할 양이 적어짐
  → 대량 INSERT 시 I/O가 분산됨
```

## 8. 메모리 사용량 분석

```
프레임 1개 = frame_t 구조체(32바이트) + data 버퍼(4096바이트)
           = 4128바이트

전체 캐시 = 256 × 4128 = 1,056,768바이트 ≈ 1MB

page_size별 메모리:
  4096 × 256  = 1MB   (x86_64)
  16384 × 256 = 4MB   (Apple Silicon)
```

### 캐시 적중률 시나리오

```
DB 전체 10,000페이지 (40MB), 프레임 256개 (1MB):

시나리오 1 — B+ tree 검색 반복:
  루트(page 5)는 거의 항상 캐시에 있음 (매번 접근)
  내부 노드도 수십 개뿐이라 대부분 캐시에 남음
  → 디스크 읽기: 리프 + 힙 = 2회/쿼리

시나리오 2 — 전체 스캔:
  페이지를 순서대로 읽으면서 캐시를 계속 교체
  적중률 낮음 → 매 페이지마다 디스크 읽기
  → 캐시의 이점이 제한적

시나리오 3 — 핫 데이터 집중:
  최근 INSERT된 행에 SELECT가 집중
  해당 힙/리프 페이지가 캐시에 남아있을 확률 높음
  → 높은 적중률
```

## 9. flush — 캐시 → 디스크 동기화

```
pager_flush_all() 호출 시 (프로그램 종료 전):

  frames[0]: valid, dirty, page_id=5  → pwrite(fd, data, 4096, 5*4096)
  frames[1]: valid, clean, page_id=3  → 건너뜀 (수정 안 됨)
  frames[2]: valid, dirty, page_id=7  → pwrite(fd, data, 4096, 7*4096)
  frames[3]: invalid                   → 건너뜀
  ...
  헤더(page 0) 기록 → pwrite(fd, header, 4096, 0)
  fsync(fd) → OS 버퍼까지 디스크에 확실히 기록

  → 모든 dirty 프레임의 is_dirty = false
  → 디스크와 메모리가 동기화됨
```

```
  └────┬───────────┬────────┘

       ▼           ▼
  ┌──────┐   ┌──────────────┐
  │pin++ │   │ evict_frame()│ ← LRU로 교체 대상 선택
  │tick  │   │ dirty? flush │
  │갱신  │   │ pread()      │ ← 디스크에서 로드
  └──┬───┘   └──────┬───────┘
     │              │
     ▼              ▼
  ┌─────────────────────────┐
  │  data 포인터 반환        │

               ▼
  ┌─────────────────────────┐
  │  dirty 워터마크 체크     │
  │  64개 이상? → 선제 flush │
  └─────────────────────────┘
```
