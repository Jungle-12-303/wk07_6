# Q17. OS·커널·스케줄러·프로세스·스레드 한번에 정리

> CSAPP 9.7 | 7부. 커널과 시스템 | 기본

## 질문

1. OS, 커널, 스케줄러의 관계를 계층적으로 설명하시오.
2. 프로세스와 스레드가 공유하는 것 / 공유하지 않는 것을 구분하시오.
3. [검증] 프로세스 A의 스레드 T1, T2가 같은 전역 변수를 수정할 때, 페이지 테이블은 몇 개이고 물리 프레임은 공유되는지 설명하시오.

## 답변

### 최우녕

> OS, 커널, 스케줄러의 관계를 계층적으로 설명하시오.

OS(운영체제)는 하드웨어를 관리하고 프로그램에 실행 환경을 제공하는 시스템 소프트웨어 전체다.
커널은 OS의 핵심 부분으로, 메모리 관리, 프로세스 관리, 파일 시스템, 디바이스 드라이버 등
하드웨어에 직접 접근하는 코드가 모여있다. 커널 모드에서만 실행된다.
스케줄러는 커널 내부의 한 모듈로, 어떤 프로세스(스레드)를 언제 CPU에 올릴지 결정한다.

PCB(Process Control Block)는 커널이 프로세스마다 하나씩 갖고 있는 구조체다.
프로세스의 모든 상태 정보가 여기 들어있다. Linux에서는 `task_struct`라는 이름으로 구현되어 있다.

PCB에 담기는 정보:
- PID (프로세스 ID)
- 레지스터 상태 (PC, SP, 범용 레지스터 등)
- CR3 값 (이 프로세스의 PML4 물리 주소)
- 프로세스 상태 (Running, Ready, Blocked 등)
- 열린 파일 디스크립터 목록
- 메모리 매핑 정보 (vm_area_struct 리스트)

Context Switch(프로세스 전환) 시 커널이 하는 일:
1. 현재 프로세스의 레지스터 상태를 현재 PCB에 저장
2. 다음 프로세스의 PCB에서 레지스터 복원
3. CR3를 다음 프로세스의 PML4 물리 주소로 교체 (PCB에서 읽음)
4. CR3가 바뀌면 TLB가 flush됨 (PCID 사용 시 부분 flush)

결국 Context Switch = "PCB를 바꿔치기"하는 것이다.

```text
  OS (운영체제 전체)
  |
  ㄴ 커널 (핵심, 커널 모드)
  |  |
  |  ㄴ 메모리 관리자 (VM, 페이지 테이블, 페이지 폴트 핸들러)
  |  ㄴ 스케줄러 (CPU 시간 배분, context switch 결정)
  |  ㄴ 파일 시스템
  |  ㄴ 디바이스 드라이버
  |
  ㄴ 시스템 유틸리티, 셸 등 (유저 모드)
```

> 프로세스와 스레드가 공유하는 것 / 공유하지 않는 것을 구분하시오.

프로세스는 독립된 주소 공간을 가진 실행 단위다. fork()로 만들면 부모와 별개의 페이지 테이블을 갖는다.
스레드는 같은 프로세스 안에서 실행 흐름만 나눈 것이다. 주소 공간을 공유한다.

```text
  공유하는 것 (같은 프로세스의 스레드끼리):
  ㄴ 코드 영역 (text)
  ㄴ 전역 변수, 힙 (data, bss, heap)
  ㄴ 열린 파일 디스크립터
  ㄴ 페이지 테이블 (같은 CR3)

  공유하지 않는 것 (스레드마다 독립):
  ㄴ 스택 (각 스레드 고유 스택)
  ㄴ 레지스터 상태 (PC, SP 등)
  ㄴ 스레드 ID (TID)
  ㄴ errno (thread-local storage)
```

> [검증] 프로세스 A의 스레드 T1, T2가 같은 전역 변수를 수정할 때, 페이지 테이블은 몇 개이고 물리 프레임은 공유되는지 설명하시오.

```text
━━━ 구조 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  프로세스 A
  |
  ㄴ 페이지 테이블: 1개 (T1, T2 공유, CR3 동일)
  |
  ㄴ T1 (스레드 1)
  |  ㄴ 고유 스택, 고유 레지스터
  |  ㄴ 전역 변수 접근: VA 0x6000 → PPN 0x30
  |
  ㄴ T2 (스레드 2)
     ㄴ 고유 스택, 고유 레지스터
     ㄴ 전역 변수 접근: VA 0x6000 → PPN 0x30

━━━ 답 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  페이지 테이블: 1개 (같은 프로세스이므로 CR3가 같음)
  물리 프레임: 공유됨 (같은 VA → 같은 페이지 테이블 → 같은 PPN)

  T1이 전역 변수에 값을 쓰면 T2가 같은 물리 주소를 읽으므로
  T2에서 변경된 값이 바로 보인다.

━━━ 문제점 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  T1과 T2가 동시에 같은 전역 변수를 수정하면 race condition 발생.
  ㄴ 해결: mutex, semaphore 등 동기화 도구 필요
  ㄴ 예: pthread_mutex_lock(&lock) → 수정 → unlock

  이게 멀티스레드 프로그래밍이 어려운 이유다.
  메모리를 공유하기 때문에 빠르지만, 동기화를 안 하면 버그가 난다.
```
