# Q21. 헤더의 하위 3비트 트릭 — 정렬이 만든 공짜 비트

> CSAPP 9.9 | 8부. malloc | 검증

## 질문

1. 블록 크기가 항상 8의 배수인 이유를 설명하시오.
2. 하위 3비트에 저장되는 플래그들을 나열하고 각각의 역할을 설명하시오.
3. [검증] 헤더값이 `0x00000019`일 때 블록 크기는? 할당 상태는? 다음 블록 헤더의 주소를 계산하시오.

## 답변

### 최우녕

> 블록 크기가 항상 8의 배수인 이유를 설명하시오.

x86-64에서 가장 큰 기본 타입은 double과 포인터로 둘 다 8바이트다.
CPU가 데이터를 읽을 때 주소가 자연 정렬(natural alignment)되어 있으면
한 번의 메모리 접근으로 읽을 수 있지만,
정렬이 안 되면 두 번 접근해야 하거나 아예 에러가 난다.

malloc이 반환하는 주소가 항상 8바이트 정렬이려면
블록 시작 주소가 8의 배수여야 한다.
블록 크기도 8의 배수로 만들면 다음 블록도 자동으로 8바이트 정렬된다.

블록 크기가 8의 배수라는 건 이진수로 하위 3비트가 항상 000이라는 뜻이다.
이 3비트는 크기 정보에 사용되지 않으므로 "공짜 비트"로 다른 용도로 쓸 수 있다.

> 하위 3비트에 저장되는 플래그들을 나열하고 각각의 역할을 설명하시오.

```text
  헤더 (4바이트 or 8바이트):
  [  블록 크기 (상위 비트)  |  P  |  -  |  A  ]
                             bit2  bit1  bit0

  bit 0 (A): 현재 블록의 할당 상태
  ㄴ 1 = 할당됨 (allocated)
  ㄴ 0 = 가용 (free)

  bit 1: 구현에 따라 다름 (예약 비트)
  ㄴ glibc에서는 mmap으로 할당된 블록인지 표시

  bit 2 (P): 이전 블록의 할당 상태
  ㄴ 1 = 이전 블록이 할당됨
  ㄴ 0 = 이전 블록이 가용
  ㄴ coalesce 시 이전 블록을 합칠 수 있는지 빠르게 판단하기 위함
```

블록 크기를 읽으려면 하위 3비트를 마스킹해서 제거한다:
`size = header & ~0x7`   (하위 3비트를 0으로 만듦)

할당 비트를 읽으려면:
`is_alloc = header & 0x1`   (bit 0만 꺼냄)

> [검증] 헤더값이 `0x00000019`일 때 블록 크기는? 할당 상태는? 다음 블록 헤더의 주소를 계산하시오.

```text
━━━ 헤더 분석 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  헤더 = 0x19 = 0b 0001 1001

  하위 3비트: 001
  ㄴ bit 0 (A) = 1 → 현재 블록: 할당됨
  ㄴ bit 1     = 0
  ㄴ bit 2 (P) = 0 → 이전 블록: 가용

  블록 크기 = 0x19 & ~0x7
            = 0x19 & 0xFFFFFFF8
            = 0x18
            = 24바이트

  확인: 0x19 = 25 → 25 - 1(하위 비트) = 24 = 8 × 3 ✓

━━━ 다음 블록 헤더 주소 ━━━━━━━━━━━━━━━━━━━━━━━━

  현재 블록 헤더 주소를 hp라 하면:

  다음 블록 헤더 = hp + 블록 크기
                 = hp + 24

  예: hp = 0x1000이면
  ㄴ 현재 블록: 0x1000 ~ 0x1017 (24바이트)
  ㄴ 다음 블록 헤더: 0x1018

━━━ 정리 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  헤더 0x19 → 블록 크기 24B, 현재 할당됨, 이전 블록 가용
  다음 블록 = 헤더 주소 + 24

  이 트릭 덕분에 헤더 하나에 크기와 상태 정보를 모두 담을 수 있다.
  별도의 상태 필드가 필요 없으니 메모리를 절약한다.
```
