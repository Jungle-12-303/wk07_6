# Q12. CR3·프로세스별 페이지 테이블·물리 페이지 공유

> CSAPP 9.6-9.7 | 4부. 주소 변환 | 심화

## 질문

1. CR3 레지스터의 역할과 context switch 시 어떻게 사용되는지 설명하시오.
2. 프로세스 A, B가 같은 공유 라이브러리를 쓸 때, 각각의 페이지 테이블과 물리 페이지가 어떻게 구성되는지 서술하시오.
3. [검증] 프로세스 A의 VA `0x7000`과 프로세스 B의 VA `0x9000`이 같은 물리 프레임을 가리킬 수 있는 이유를 설명하시오.

## 답변

### 최우녕

> CR3 레지스터의 역할과 context switch 시 어떻게 사용되는지 설명하시오.

CR3는 현재 실행 중인 프로세스의 PML4(최상위 페이지 테이블)의 물리 주소를 담는 레지스터다.
MMU가 페이지 테이블 워크를 시작할 때 CR3에서 PML4 베이스를 읽어오므로,
CR3 값이 곧 "어떤 프로세스의 주소 공간을 쓸 것인가"를 결정한다.

Context Switch(프로세스 전환) 시 커널이 하는 일:
1. 현재 프로세스의 레지스터 상태를 PCB에 저장
2. 다음 프로세스의 PCB에서 레지스터 복원
3. **CR3를 다음 프로세스의 PML4 물리 주소로 교체**
4. CR3가 바뀌면 TLB가 flush됨 (PCID 사용 시 부분 flush)

CR3 하나만 바꾸면 같은 VA라도 완전히 다른 PA로 변환된다.
프로세스 A의 0x4000과 프로세스 B의 0x4000이 서로 다른 물리 메모리를 가리키는 이유다.

> 프로세스 A, B가 같은 공유 라이브러리를 쓸 때, 각각의 페이지 테이블과 물리 페이지가 어떻게 구성되는지 서술하시오.

프로세스 A와 B는 각각 독립된 페이지 테이블을 갖는다. CR3가 다르니까.
하지만 공유 라이브러리(예: libc.so)의 코드 페이지는 DRAM에 한 번만 올리고,
A의 페이지 테이블과 B의 페이지 테이블이 둘 다 같은 물리 프레임을 가리킨다.

이렇게 하는 이유는 libc 같은 라이브러리의 코드 영역은 read-only이기 때문이다.
읽기만 하니까 누가 읽어도 내용이 바뀌지 않는다.
같은 데이터를 프로세스마다 DRAM에 따로 올리면 메모리 낭비다.
커널이 A의 PTE와 B의 PTE에 같은 PPN을 적어주면
물리 메모리 한 장으로 여러 프로세스가 같이 쓸 수 있다.

만약 데이터 영역(read-write)이라면 한쪽이 수정할 수 있으므로
COW(Copy-on-Write)로 쓰기 시점에 복사본을 만들어 분리한다.

```text
  프로세스 A 페이지 테이블          물리 메모리
  VA 0x7000 → PPN 0x50 ──────→ [ libc 코드 페이지 (read-only) ]
                                      ↑
  프로세스 B 페이지 테이블            │
  VA 0x9000 → PPN 0x50 ──────────────┘

  ※ 같은 PPN = 같은 DRAM 셀
     코드가 read-only이므로 누가 읽어도 안전
     DRAM에 한 벌만 올리면 되니까 메모리 절약
```

> [검증] 프로세스 A의 VA `0x7000`과 프로세스 B의 VA `0x9000`이 같은 물리 프레임을 가리킬 수 있는 이유를 설명하시오.

```text
━━━ 왜 가능한가 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  핵심: VA → PA 변환은 프로세스별 페이지 테이블이 결정한다.
  VA가 같을 필요도 없고, PPN만 같으면 같은 물리 메모리다.

━━━ 구체적 흐름 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  프로세스 A 실행 중 (CR3 = A의 PML4 물리 주소)
  |
  CPU: VA 0x7000 → MMU
  ㄴ VPN = 0x7000 >> 12 = 0x7
  ㄴ A의 페이지 테이블 워크 → PTE에서 PPN = 0x50
  ㄴ PA = 0x50000

  프로세스 B 실행 중 (CR3 = B의 PML4 물리 주소)
  |
  CPU: VA 0x9000 → MMU
  ㄴ VPN = 0x9000 >> 12 = 0x9
  ㄴ B의 페이지 테이블 워크 → PTE에서 PPN = 0x50
  ㄴ PA = 0x50000

  ※ VA는 다르지만 (0x7000 vs 0x9000)
     PPN이 같으므로 (0x50) 같은 물리 프레임에 접근

━━━ 언제 이렇게 되나 ━━━━━━━━━━━━━━━━━━━━━━━━━━

  ㄴ 공유 라이브러리: libc 등의 코드 페이지 (read-only)
  ㄴ fork() 직후: 부모-자식이 COW로 같은 프레임 공유
  ㄴ mmap MAP_SHARED: 여러 프로세스가 같은 파일을 매핑

  커널이 두 프로세스의 PTE에 같은 PPN을 기록하면 성립한다.
  페이지 테이블은 프로세스별로 독립이지만,
  가리키는 물리 프레임은 공유할 수 있다.
```
