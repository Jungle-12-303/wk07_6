# Q3. 페이지 크기 4KB의 근거 — Goldilocks 원리

> CSAPP 9.3 | 1부. 주소와 공간 | 심화

## 질문

1. 페이지 크기가 너무 작을 때/클 때의 단점을 각각 서술하시오.
2. 4KB가 "적당한" 이유를 내부 단편화, 페이지 테이블 크기, 디스크 I/O 관점에서 설명하시오.
3. [검증] 64-bit 시스템, 페이지 크기 4KB일 때 단일 페이지 테이블의 최대 엔트리 수와 테이블 크기를 계산하시오.

## 답변

페이지가 너무 작으면 페이지 테이블 크기와 번역 관리 비용이 커지고, disk I/O가 너무 잘게 쪼개져 비효율적임.
페이지가 너무 크면 내부 단편화와 과잉 적재(overfetch) 가 커지며, 실제로 안 쓰는 데이터까지 함께 올려 locality 활용이 나빠질 수 있음.
개념적으로 총비용은 C(P)=A/P + B·P 로 볼 수 있고, 이를 미분하면 dC/dP = -A/P² + B = 0, 따라서 균형점은 P* = √(A/B) 임.

64-bit 시스템에서 4KB page면 virtual page 수는 2^64 / 2^12 = 2^52, 
PTE가 8B라면 단일 페이지 테이블 크기는 2^52 × 8 = 2^55B = 32PB 가 되어 비현실적이므로 Multi-Level Page Table 이 필요함.

현대 시스템은 4KB만 고집하지 않고, x86-64는 보통 4KB base page를 유지하는 반면 arm64는 4KB·16KB·64KB base page 를 지원하는 등, 
TLB reach·페이지 테이블 오버헤드·워크로드 특성에 따라 더 큰 base page를 쓰는 방향도 함께 발전 중임.

키워드: Goldilocks principle, C(P)=A/P + B·P, P*=√(A/B), internal fragmentation, overfetch, page table size, 32PB, Multi-Level Page Table, TLB reach, 4KB, 16KB, 64KB, x86-64, arm64

### 최우녕

> 페이지 크기가 너무 작을 때/클 때의 단점을 각각 서술하시오.

페이지 크기가 너무 작으면 페이지 수가 폭발적으로 늘어나서
페이지 테이블 엔트리를 기록하기 위한 메모리를 많이 써야 한다.
TLB에 담을 수 있는 커버리지도 줄어들어 TLB 미스가 잦아지고,
디스크 I/O도 작은 단위로 여러 번 해야 해서 비효율적이다.

페이지 크기가 너무 크면 내부 단편화가 심해진다.
예를 들어 1바이트만 필요한데 2MB 페이지를 할당하면 나머지가 전부 낭비된다.
페이지 폴트 시 디스크에서 읽어야 할 양도 커져서 폴트 처리 시간이 늘어난다.

> 4KB가 "적당한" 이유를 내부 단편화, 페이지 테이블 크기, 디스크 I/O 관점에서 설명하시오.

페이지의 크기는 하드웨어에 따라 다르기 때문에 4KB가 절대적으로 적당하다고 말할 수는 없다.
(Apple M4는 16KB를 쓴다.) 다만 x86-64 기준으로 4KB가 균형 잡힌 이유는:

내부 단편화: 페이지는 VA↔PA 매핑 단위이면서 디스크↔DRAM I/O 단위이기도 한데,
이 단위 이하의 데이터를 빈번하게 할당하면 채워지지 않는 남은 공간이 낭비된다.
4KB면 평균 낭비가 약 2KB 정도로, 대부분의 프로그램에서 감당 가능한 수준이다.

페이지 테이블 크기: 48비트 VA에서 오프셋 12비트를 빼면 VPN이 36비트다.
페이지 테이블은 VPN을 인덱스로 PPN을 찾는 구조이므로,
단일 테이블(배열)로 만들면 2^36개(약 687억 개) 엔트리를 전부 메모리에 갖고 있어야 한다.
PTE 하나가 8바이트이므로 2^36 × 8B = 512GB가 된다.
DRAM이 24GB인 내 맥에서 프로세스 하나의 테이블에 512GB는 말이 안 된다.

단일 테이블이 이렇게 큰 이유는, 배열이기 때문에 빈칸도 전부 존재해야 하기 때문이다.
실제 프로세스가 쓰는 메모리는 수 MB ~ 수 GB인데 687억 칸 대부분은 비어있는 낭비다.

그래서 다단계(4단계) 페이지 테이블을 쓴다.
36비트 VPN을 9비트씩 4등분해서 폴더 구조처럼 트리로 만든다.
각 단계 테이블은 2^9 = 512개 엔트리, 512 × 8B = 4KB → 딱 물리 페이지 1장에 들어간다.
핵심은 사용하지 않는 VA 영역은 해당 단계 테이블을 아예 만들지 않는다는 것이다(NULL).
PML4의 엔트리 하나가 512GB 영역을 커버하는데, NULL로 두면 그 아래 모든 테이블이 생략된다.
결과적으로 실제 메모리 사용량은 프로세스가 쓰는 영역에 비례해 수 KB ~ 수 MB로 줄어든다.

디스크 I/O: 디스크의 최소 접근 시간은 수 ms로 매우 비싸다.
4KB면 한 번 폴트 시 충분한 양을 올려 spatial locality를 살리면서도
불필요하게 큰 전송으로 시간을 낭비하지 않는 균형점이다.

> [검증] 64-bit 시스템, 페이지 크기 4KB일 때 단일 페이지 테이블의 최대 엔트리 수와 테이블 크기를 계산하시오.

```text
전제: x86-64, 48비트 가상 주소, 4KB 페이지, PTE 1개 = 8바이트

━━━ 1. 단일 테이블 계산 ━━━━━━━━━━━━━━━━━━━━━━━━

  VA 48비트 - 오프셋 12비트(4KB) = VPN 36비트

  최대 엔트리 수 = 2^36 = 68,719,476,736 (약 687억 개)
  테이블 크기    = 2^36 × 8B = 2^39B = 512GB

  ※ DRAM 24GB인 맥에서 테이블 하나가 512GB → 불가능

━━━ 2. 왜 불가능한가 ━━━━━━━━━━━━━━━━━━━━━━━━━━━

  48비트 주소 공간 = 256TB
  실제 프로세스가 쓰는 메모리 = 수 MB ~ 수 GB
  687억 개 엔트리 중 대부분은 "사용 안 함"

  ㄴ 빈 엔트리를 위해 512GB를 쓰는 건 낭비

━━━ 3. 해결: 다단계 페이지 테이블 ━━━━━━━━━━━━━━

  36비트 VPN을 9비트씩 4등분:

  bits[47:39]  bits[38:30]  bits[29:21]  bits[20:12]
     PML4         PDPT         PD           PT
    (9bit)       (9bit)       (9bit)       (9bit)

  각 단계 테이블:
    2^9 = 512개 엔트리 × 8B = 4096B = 4KB = 딱 1페이지

━━━ 4. 실제 구조 예시 ━━━━━━━━━━━━━━━━━━━━━━━━━━

  PML4 (항상 1개, 4KB)
  |
  ㄴ [0]   → PDPT 할당 (이 영역 사용 중)
  |          |
  |          ㄴ [0] → PD 할당
  |          |        |
  |          |        ㄴ [0] → PT 할당 (4KB)
  |          |                  |
  |          |                  ㄴ [4] → PPN = 0xA3
  |          |                  ㄴ 나머지 → 0 (미사용)
  |          |
  |          ㄴ [1~511] → NULL (안 만듦)
  |
  ㄴ [1]   → NULL (이 영역 안 씀 → PDPT 자체가 없음)
  ㄴ [2]   → NULL
  ㄴ ...
  ㄴ [511] → NULL

  NULL인 곳은 하위 테이블을 아예 할당하지 않는다
  ㄴ 안 쓰는 256TB 영역의 테이블 비용 = 0

━━━ 5. 비교 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  단일 테이블:  항상 512GB (사용 여부와 무관)
  다단계 테이블: 쓰는 만큼만 (수 KB ~ 수 MB)

  4KB 페이지가 깔끔한 이유:
  ㄴ 9비트 × 4단계 = 36비트 VPN을 정확히 소화
  ㄴ 각 테이블이 4KB = 물리 페이지 1장에 딱 맞음
  ㄴ 할당/해제를 페이지 단위로 관리 가능
```

### 최영빈

> 1. 페이지의 크기가 너무 작으면 디스크에 접근해야하는 횟수가 많아져 성능 저하를 유발할 수 있다. 페이지의 크기가 너무 크면 디스크에서 불필요한 데이터를 포함해서 가져올 가능성이 높아진다.
