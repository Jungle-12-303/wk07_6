# Q24. free()의 설계 철학 — 왜 에러를 알려주지 않는가?

> CSAPP 9.9 | 8부. malloc | 심화

## 질문

1. C 표준이 free(invalid)를 UB로 정의한 이유를 설명하시오.
2. 잘못된 포인터를 free 했을 때 일어날 수 있는 일을 3가지 이상 서술하시오.
3. [검증] 다음 각 경우에 어떤 문제가 발생하는지 설명하시오: (a) double free (b) free(p+1) (c) int x; free(&x)

## 답변

### 최우녕

> C 표준이 free(invalid)를 UB로 정의한 이유를 설명하시오.

free()는 malloc이 반환한 포인터만 받도록 설계되어 있다.
잘못된 포인터를 검증하려면 free가 호출될 때마다
"이 주소가 정말 malloc으로 할당된 블록의 시작 주소인가"를 확인해야 하는데,
이걸 매번 하면 성능이 크게 떨어진다.

C의 설계 철학은 "프로그래머를 신뢰하고, 런타임 오버헤드를 최소화한다"이다.
그래서 잘못된 포인터로 free를 호출하면 에러를 알려주는 대신
Undefined Behavior(정의되지 않은 동작)로 남겨뒀다.
검증 비용을 프로그래머의 책임으로 넘긴 것이다.

free()의 반환값이 void인 것도 같은 이유다.
성공/실패를 알려주려면 매번 검증을 해야 하는데, C는 그 비용을 거부한 거다.

> 잘못된 포인터를 free 했을 때 일어날 수 있는 일을 3가지 이상 서술하시오.

1. 힙 메타데이터 파괴: free는 포인터 앞에 있는 헤더(블록 크기, 할당 비트)를 읽는다.
   잘못된 주소의 헤더를 읽으면 엉뚱한 메모리를 해제하거나 free list가 깨진다.

2. 다른 블록 침범: 깨진 free list 때문에 다음 malloc이 이미 사용 중인 블록을
   반환할 수 있다. 두 포인터가 같은 메모리를 가리키게 되어 데이터가 덮어씌워진다.

3. Segmentation Fault: 잘못된 주소가 프로세스에 할당되지 않은 VA 영역이면
   MMU가 페이지 테이블을 뒤져도 해당 PTE가 invalid(커널이 할당한 적 없음)이다.
   MMU는 "이 주소 변환 못 함"이라고 CPU에 예외를 보내고,
   커널 폴트 핸들러가 "복구 불가능한 접근"이라고 판단하면
   프로세스에 SIGSEGV(Segmentation Fault) 시그널을 보내서 강제 종료한다.
   q04의 Invalid 상태(PTE 3번째 상태)와 같은 경우다.

4. 보안 취약점: 공격자가 힙 메타데이터를 의도적으로 조작하면
   임의 주소에 쓰기(arbitrary write)가 가능해져서 코드 실행 공격으로 이어질 수 있다.

> [검증] 다음 각 경우에 어떤 문제가 발생하는지 설명하시오: (a) double free (b) free(p+1) (c) int x; free(&x)

```text
━━━ 먼저: malloc 블록의 구조 이해 ━━━━━━━━━━━━━━

  malloc(100) 하면 실제 메모리는 이렇게 생겼다:

  [ 헤더 (8B) ][ 사용자 데이터 (100B) ]
       ↑              ↑
   블록 크기/할당비트    malloc이 반환한 포인터 p

  free(p) 하면 free는 p 바로 앞 8바이트를 헤더로 읽어서
  "이 블록이 몇 바이트짜리구나"를 알아내고 free list에 넣는다.
  이 구조를 모르면 아래 3가지가 왜 위험한지 이해가 안 된다.

━━━ (a) double free ━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  char *p = malloc(100);
  free(p);     // 1. 블록을 free list에 넣음
  free(p);     // 2. 같은 블록을 또 넣음

  free list는 "여기 빈 블록 있어요" 목록이다.
  같은 블록이 두 번 등록되면:

  ㄴ 다음에 malloc을 두 번 호출하면 같은 주소를 두 번 반환함
  ㄴ char *a = malloc(100);  // p의 주소 반환
  ㄴ char *b = malloc(100);  // 또 p의 주소 반환
  ㄴ a와 b가 같은 메모리를 가리킴
  ㄴ a에 쓰면 b의 데이터가 덮어씌워짐 → 원인 찾기 매우 어려운 버그

  방지: free 후 반드시 p = NULL; 하면 두 번째 free(NULL)은 아무것도 안 함

━━━ (b) free(p+1) ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  char *p = malloc(100);
  free(p + 1);  // 블록 시작이 아닌 1바이트 뒤 주소

  free는 전달받은 주소 바로 앞 8바이트를 헤더로 읽는다.
  근데 p+1 앞 8바이트에는 헤더가 아니라 사용자 데이터가 있다.

  [ 헤더 ][ 데이터 ... ]
           ↑ p  ↑ p+1
                 free가 여기 앞 8B를 헤더로 착각

  ㄴ 엉뚱한 값을 "블록 크기 = 3억 바이트"처럼 해석할 수 있음
  ㄴ 힙의 free list 구조 전체가 깨짐
  ㄴ 이후 malloc/free 모두 비정상 동작
  ㄴ 바로 크래시하면 차라리 낫고, 조용히 데이터가 오염되면 더 위험

━━━ (c) int x; free(&x) ━━━━━━━━━━━━━━━━━━━━━━━

  int x = 42;
  free(&x);  // 스택 변수의 주소를 free

  메모리 영역은 크게 스택, 힙, 코드, 데이터로 나뉜다.
  malloc은 힙 영역만 관리한다. &x는 스택 영역의 주소다.

  free는 &x 앞 8바이트를 헤더로 읽으려 하지만
  스택에는 malloc 헤더가 없다. 거기엔 다른 지역 변수나
  함수 리턴 주소가 있다.

  ㄴ 스택 메모리를 힙 블록으로 착각하고 free list에 등록
  ㄴ 다음 malloc이 스택 영역을 "빈 힙 블록"으로 반환할 수 있음
  ㄴ 스택과 힙이 뒤섞이면 함수 리턴 주소가 덮어씌워져서 크래시
  ㄴ 공격자가 악용하면 원하는 코드를 실행할 수도 있음

━━━ 공통 교훈 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  free()는 "이 포인터가 올바른가"를 검증하지 않는다.
  전달받은 주소 앞을 무조건 헤더로 읽기 때문에
  잘못된 주소를 넘기면 힙 전체가 오염될 수 있다.

  프로그래머가 지켜야 할 규칙:
  ㄴ malloc이 반환한 원래 주소로만 free
  ㄴ free 후 포인터를 NULL로 설정
  ㄴ 스택/전역 변수 주소를 절대 free하지 않기
```
