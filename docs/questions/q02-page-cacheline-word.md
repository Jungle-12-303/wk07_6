# Q2. 페이지·캐시라인·워드 — 계층이 다르다

> CSAPP 9.1 | 1부. 주소와 공간 | 기본

## 질문

1. 페이지(4KB), 캐시라인(64B), 워드(4/8B) 각각이 속한 메모리 계층을 설명하시오.
2. 세 단위의 크기가 다른 이유를 locality 관점에서 서술하시오.
3. [검증] CPU가 주소 `0x1234`의 int를 읽을 때, 페이지·캐시라인·워드 단위로 어떤 일이 일어나는지 순서대로 설명하시오.

## 답변

### 최우녕

> 페이지(4KB), 캐시라인(64B), 워드(4/8B) 각각이 속한 메모리 계층을 설명하시오.

워드(Word)는 CPU가 레지스터에서 한 번에 처리하는 데이터 단위다.
x86-64에서 1워드 = 8바이트이며, int(4B)는 반워드, 포인터(8B)는 1워드에 해당한다.
우리가 사용하는 malloc, realloc, calloc은 이 워드의 배수(보통 2워드 = 16B)로 블록을 할당하게 되고,
예를 들어 char 하나를 위해 1바이트만 요청해도 최소 블록 크기만큼 할당되어 나머지가 낭비된다.
이렇게 블록 정렬 때문에 생기는 낭비를 내부 단편화(패딩 바이트)라 한다.

캐시라인(Cache Line)은 CPU 캐시(L1/L2/L3)와 DRAM 사이의 전송 단위다.
CPU가 1바이트만 필요하더라도 캐시라인 크기(x86-64에서 64B)를 통째로 캐시에 올린다.
인접한 데이터를 미리 가져와서 spatial locality를 활용하기 위한 것이다.

페이지(Page)는 가상 메모리 주소와 물리 메모리 주소를 연결하는 블록 단위다.
운영체제가 페이지 단위로 VA→PA 변환을 관리하며,
디스크↔DRAM 사이의 스왑도 페이지 단위로 이루어진다.

```text
계층 구조:

  CPU 레지스터   ←  워드 (4/8B)      : 연산 단위
       ↕
  L1/L2/L3 캐시  ←  캐시라인 (64B)   : 캐시↔DRAM 전송 단위
       ↕
  DRAM           ←  페이지 (4KB)     : DRAM↔디스크 전송 단위
       ↕
  디스크(스왑)
```

※ 이 세 단위는 CPU 아키텍처마다 다르다.

```text
  항목         x86-64 (Intel/AMD)    ARM64 (Apple M4)
  ─────────    ──────────────────    ─────────────────
  워드         8B                    8B
  캐시라인     64B                   128B
  페이지       4KB                   16KB
```

Apple Silicon(M1~M4)은 캐시라인이 128B로 x86-64의 2배이고,
페이지도 16KB로 4배 크다. 같은 코드라도 TLB 커버리지, 캐시 미스 특성,
내부 단편화 양상이 달라지기 때문에 아키텍처를 알고 있어야 한다.

실측 (현재 내 맥):
- `sysctl hw.pagesize` → 16384 (16KB)
- `sysctl hw.cachelinesize` → 128 (128B)
- `python3 -c "import struct; print(struct.calcsize('P'))"` → 8 (8B, 64비트)
- `uname -m` → arm64 (Apple M4)

> 세 단위의 크기가 다른 이유를 locality 관점에서 서술하시오.

페이지, 캐시라인, 워드는 운영체제와 CPU마다 차이가 있으며,
세 단위의 크기가 다른 핵심 이유는 각 계층의 miss penalty(빗나갔을 때 비용)가 다르기 때문이다.

워드(4/8B): 레지스터↔캐시는 ~4사이클로 빠르다.
CPU가 필요한 정확한 크기만 요청하면 충분하다.

캐시라인(64B): 캐시 미스 시 DRAM 접근에 ~200사이클이 든다.
한 번 비용을 치를 때 인접 데이터를 64B 묶어서 가져오면
spatial locality 덕분에 이후 접근이 HIT가 될 확률이 높다.

페이지(4KB): 페이지 폴트 시 디스크 접근에 ~10ms(수백만 사이클)가 든다.
극단적으로 비싼 비용이므로 4KB를 통째로 올려서
이후 해당 영역 접근 시 추가 폴트를 최소화한다.

정리하면 miss penalty가 클수록 블록을 크게 잡아서
한 번의 비용으로 locality 이득을 최대화하는 구조다.

> [검증] CPU가 주소 `0x1234`의 int를 읽을 때, 페이지·캐시라인·워드 단위로 어떤 일이 일어나는지 순서대로 설명하시오.

```text
전제: x86-64, 4KB 페이지, 64B 캐시라인, int = 4바이트

━━━ 1. VA 분해 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  VA = 0x1234

  페이지 관점:
  [  VPN = 0x1   |  VPO = 0x234  ]
    페이지 번호      페이지 내 오프셋

  캐시라인 관점 (VPO 0x234 = 564바이트):
    564 / 64 = 8번째 라인, 오프셋 = 564 % 64 = 52

  워드 관점:
    라인 내 오프셋 52에서 4바이트(int) 읽기

━━━ 2. 페이지 단위 — VA → PA 변환 ━━━━━━━━━━━━━━

  MMU: VPN=1 로 TLB 조회
  |
  ㄴ HIT  → PPN 획득
  ㄴ MISS → 페이지 테이블 워크 → PPN 획득
             (Valid=0이면 Page Fault → 디스크에서 4KB 통째로 로드)

  PA = (PPN << 12) | 0x234

━━━ 3. 캐시라인 단위 — 캐시 조회 ━━━━━━━━━━━━━━━

  PA에서 캐시 인덱스/태그 추출 → L1 캐시 탐색
  |
  ㄴ HIT  → 해당 64B 라인이 이미 캐시에 있음
  ㄴ MISS → DRAM에서 0x1234를 포함하는 64B 정렬 블록을 통째로 로드

━━━ 4. 워드 단위 — 데이터 추출 ━━━━━━━━━━━━━━━━━

  캐시라인 64B 중 오프셋 52 위치에서
  4바이트(int)를 꺼내 CPU 레지스터에 전달

━━━ 요약 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  페이지(4KB)  : VA→PA 변환의 단위. 폴트 시 디스크에서 4KB 로드
  캐시라인(64B): 캐시↔DRAM 전송 단위. 미스 시 64B 로드
  워드(4B)     : CPU가 실제로 사용하는 int 크기. 라인 안에서 추출
```

### 최영빈

> 1. VM, PM가 페이지단위를 사용한다. 워드는 CPU가 한번에 처리할 수 있는 정보의 크기 단위이다.
