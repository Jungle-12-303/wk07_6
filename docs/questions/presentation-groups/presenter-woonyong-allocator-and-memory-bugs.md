# 발표자료

## 이 발표가 답하려는 질문

이 발표는 다음 질문들에 답하기 위한 발표이다.

- 왜 힙에 빈 공간이 남아 있어도 `malloc()`이 실패하는가
- allocator는 빈 블록을 어떤 구조로 관리하는가
- 블록 크기와 상태 정보는 어디에 저장되는가
- `split`, `coalesce`, `placement policy`는 왜 필요한가
- `free()`는 왜 잘못된 포인터를 막아주지 않는가
- 왜 C에서는 메모리 버그가 구조적으로 자주 발생하는가

이 질문들을 하나로 묶으면 다음 문장으로 정리할 수 있다.

**Heap allocator는 단편화 문제를 줄이기 위해 복잡한 자료구조와 정책을 사용하고, C는 그 과정의 안전성 검사를 거의 프로그래머 책임으로 남겨두기 때문에 메모리 버그가 쉽게 발생한다.**

## 정리된 설명과 근거

### 1. 결론: allocator가 먼저 풀어야 하는 문제는 단편화이다

Q19의 핵심은 메모리 부족보다 메모리 조각화가 먼저 문제라는 점이다.

- 내부 단편화는 할당된 블록 내부의 낭비이다.
- 외부 단편화는 총 빈 공간은 충분하지만 연속된 큰 블록이 없어서 실패하는 상태이다.

근거는 다음과 같다.

```text
내부 단편화 예시
요청: 13B
할당: 16B
낭비: 3B
```

```text
외부 단편화 예시
[16B 할당][48B 가용][16B 할당][32B 가용]

총 가용 공간 = 48 + 32 = 80B
요청 크기 = 60B

48B 블록 -> 60B보다 작음
32B 블록 -> 60B보다 작음
중간 16B 할당 블록 때문에 두 가용 블록을 합칠 수 없음

결론: 총량은 충분하지만 연속된 60B 블록이 없어서 malloc(60)은 실패
```

즉 allocator가 풀어야 하는 핵심은 "메모리가 남아 있느냐"가 아니라 "사용 가능한 형태로 남아 있느냐"이다.

### 2. 결론: allocator는 크기별 관리 구조로 탐색 비용을 줄인다

Q20의 핵심은 allocator가 모든 블록을 순회하는 방식으로는 버틸 수 없다는 점이다.

실제 답의 비교는 다음과 같다.

```text
Implicit Free List  : O(전체 블록 수)
Explicit Free List  : O(가용 블록 수)
Segregated Free List: 해당 bin만 탐색 -> ~O(1)
```

이 결론은 처리 과정 예시로도 확인된다.

```text
bin[0]: 1~8B
bin[1]: 9~16B
bin[2]: 17~32B

malloc(12)
-> 12B에 맞는 bin[1]부터 탐색
-> bin[1]에 없으면 bin[2]에서 더 큰 블록을 가져와 split
-> 예: 32B 블록을 꺼내 16B 할당 + 16B 재삽입
```

즉 allocator는 빈 블록을 한 덩어리로 관리하지 않고,
요청 크기에 맞는 후보군만 빠르게 찾도록 설계되어 있다.

### 3. 결론: allocator는 헤더 하나에 크기와 상태를 함께 압축 저장한다

Q21의 핵심은 메타데이터 저장 비용을 최소화하는 방식이다.

실제 답의 이유는 다음과 같다.

```text
블록 크기를 8의 배수로 맞추면 하위 3비트는 항상 000이다.
따라서 하위 3비트는 크기 정보에 쓰이지 않는 공짜 비트가 된다.
```

이 공짜 비트는 다음처럼 활용된다.

```text
[ 블록 크기 | P | - | A ]

A(bit 0): 현재 블록 할당 여부
P(bit 2): 이전 블록 할당 여부
```

실제 검증 예시는 다음과 같다.

```text
헤더값 = 0x19 = 0b00011001

size = 0x19 & ~0x7 = 0x18 = 24B
A bit = 1 -> 현재 블록은 할당됨
P bit = 0 -> 이전 블록은 가용

현재 헤더 주소가 hp라면
다음 블록 헤더 주소 = hp + 24
```

즉 allocator는 별도 상태 구조체를 두는 것이 아니라,
정렬 규칙을 이용해 헤더 하나에 크기와 상태를 함께 저장한다.

### 4. 결론: split, coalesce, placement policy가 단편화와 성능을 동시에 결정한다

Q22와 Q23의 핵심은 allocator가 실제 할당과 해제 순간마다 정책 결정을 한다는 점이다.

먼저 split은 아무 때나 하는 것이 아니라 조건이 있다.

```text
split 조건
가용 블록 크기 - 요청 블록 크기 >= 최소 블록 크기

예1) 64B - 24B = 40B >= 16B -> split 가능
예2) 32B - 24B = 8B < 16B -> split 불가, 32B 전체 할당
```

즉 split을 못 하면 내부 단편화가 생기고,
split을 자주 하면 반대로 외부 단편화가 커질 수 있다.

coalesce도 실제 답에서 4가지 경우로 정리된다.

```text
Case 1: PREV 할당, NEXT 할당 -> 합치지 않음
Case 2: PREV 할당, NEXT 가용 -> 현재 + 다음 합침
Case 3: PREV 가용, NEXT 할당 -> 이전 + 현재 합침
Case 4: PREV 가용, NEXT 가용 -> 이전 + 현재 + 다음 모두 합침
```

이 판단이 빠른 이유도 실제 답에 나와 있다.

```text
다음 블록 상태 -> 다음 블록 헤더의 할당 비트 확인
이전 블록 상태 -> 현재 블록 헤더의 P비트 확인
이전 블록 크기 -> 이전 블록 풋터에서 확인

따라서 coalesce 4가지 케이스는 모두 O(1)에 처리 가능
```

배치 정책 차이도 실제 계산으로 증명된다.

```text
가용 리스트: [32B, 64B, 24B, 128B]
요청: malloc(20) -> 실제 필요 크기 24B

First Fit -> 32B 선택 -> 잔여 8B -> split 불가 -> 내부 단편화 8B
Best Fit  -> 24B 선택 -> 잔여 0B -> 단편화 없음
```

즉 allocator의 성능과 단편화는 자료구조만이 아니라
어떤 블록을 고르고, 언제 쪼개고, 언제 합치는지에 의해 결정된다.

### 5. 결론: C의 free는 성능을 위해 포인터의 올바름을 신뢰한다

Q24의 핵심은 `free()`가 안전성 검사를 충분히 하지 않는다는 점이다.

그 이유는 실제 답에서 명확하다.

```text
잘못된 포인터인지 검사하려면
"이 주소가 정말 malloc이 반환한 블록 시작 주소인가"
를 free 호출마다 확인해야 한다.

C는 이 비용을 감수하지 않고
잘못된 free를 Undefined Behavior로 남겨둔다.
```

왜 위험한지는 메모리 구조를 보면 바로 드러난다.

```text
[ 헤더 ][ 사용자 데이터 ]
   ^          ^
   |          p

free(p)는 p 바로 앞의 헤더를 읽어 블록 크기와 상태를 해석한다.
```

따라서 다음 예시들이 모두 위험해진다.

```text
double free
-> 같은 블록이 free list에 두 번 들어감
-> 이후 malloc 두 번이 같은 주소를 반환할 수 있음

free(p + 1)
-> 사용자 데이터를 헤더로 착각해서 읽음
-> 엉뚱한 크기/상태 해석
-> free list 파괴

int x;
free(&x)
-> 스택 주소를 힙 블록처럼 해석
-> 힙과 스택 정보가 뒤섞일 수 있음
```

즉 `free()`가 빠른 이유는 안전해서가 아니라,
호출자가 올바른 포인터를 넘긴다고 가정하기 때문이다.

### 6. 결론: 메모리 버그는 allocator 모델이 깨졌을 때 나타나는 구조적 결과이다

Q25의 핵심은 메모리 버그가 서로 따로 노는 실수 목록이 아니라는 점이다.

실제 답의 중요한 연결은 다음과 같다.

```text
Buffer Overflow
-> 다음 데이터나 다음 블록 헤더를 덮어씀

Use-After-Free
-> 이미 free list에 들어간 블록을 다시 사용
-> 새로 할당된 다른 데이터 오염 가능

Double Free
-> 같은 블록이 free list에 두 번 등록됨

Memory Leak
-> 할당된 메모리를 가리키는 포인터를 잃어버림

Uninitialized Read
-> malloc은 메모리를 0으로 초기화하지 않음
-> 이전 데이터 흔적을 읽을 수 있음
```

실제 검증 예시도 이 결론을 뒷받침한다.

```c
int **A = malloc(n * sizeof(int));
// ... 사용 ...
A[n] = NULL;
```

위 코드는 두 군데에서 구조를 깨뜨린다.

```text
버그 1
sizeof(int)로 할당 -> 실제 필요한 sizeof(int*)보다 작게 할당
-> 뒤쪽 원소 접근 시 버퍼 오버플로우

버그 2
A[n] 접근 -> 유효 범위는 0 ~ n-1
-> 배열 경계 밖 쓰기
-> 다음 블록 헤더를 덮어쓸 수 있음
```

즉 메모리 버그는 단순히 "코드를 잘못 썼다"에서 끝나는 문제가 아니라,
allocator가 신뢰하는 경계, 헤더, free list 구조를 깨뜨리기 때문에 치명적이다.

## 그러므로 알 수 있는 것

- allocator의 핵심 문제는 메모리 부족이 아니라 단편화 관리이다.
- allocator의 핵심 해법은 크기별 free list, 압축된 헤더, split/coalesce, 배치 정책이다.
- C는 이 구조를 빠르게 유지하기 위해 검사를 최소화한다.
- 그래서 메모리 버그는 단순 실수가 아니라 구조적으로 발생하기 쉬운 위험이다.

## 발표 마무리용 정리

이 발표를 통해 알 수 있는 것은,
heap allocator는 단순히 메모리를 빌려주는 함수 묶음이 아니라
단편화를 줄이고 성능을 유지하기 위한 자료구조와 정책의 집합이라는 점이다.

그리고 C에서는 이 복잡한 시스템을 안전하게 감싸주지 않기 때문에,
잘못된 포인터 하나와 잘못된 메모리 접근 하나가 곧바로 메모리 버그로 이어질 수 있다.

즉 **allocator를 이해해야 메모리 버그도 이해할 수 있다.**

# 발표 4. Heap allocator와 메모리 버그

## 이 발표의 역할

이 파트는 앞선 발표들이 만든 하드웨어, 가상 메모리, 커널의 토대 위에서, 개발자가 직접 맞닥뜨리는 사용자 공간 allocator와 메모리 버그를 설명하는 역할이다.
즉 "프로그램이 heap을 어떻게 쓰고, 왜 느려지고, 왜 위험해지는가"를 다룬다.

## 한 줄 흐름

`fragmentation` -> `segregated free list` -> `block header bit trick` -> `split / coalesce` -> `placement policy` -> `free() 설계` -> `memory bugs`

   - [검증] 힙에 [16B할당][48B가용][16B할당][32B가용] 상태에서 malloc(60)이 실패하는 이유를 내부/외부 단편화로 설명하시오.

   <details>
   <summary>최우녕 답변</summary>

   > 내부 단편화와 외부 단편화의 차이를 정의하시오.

   내부 단편화(Internal Fragmentation):
   할당된 블록 안에서 실제로 쓰지 않는 부분이 낭비되는 것이다.
   예를 들어 13바이트를 요청했는데 정렬 때문에 16바이트 블록을 할당하면
   3바이트가 패딩으로 낭비된다. 블록 "안쪽"의 낭비라서 내부 단편화다.

   외부 단편화(External Fragmentation):
   빈 공간의 총합은 충분하지만 연속되지 않아서 큰 블록을 할당할 수 없는 상태다.
   예를 들어 48B짜리 빈 공간과 32B짜리 빈 공간이 따로 있으면
   합치면 80B인데 60B짜리 연속 블록은 만들 수 없다.
   블록 "바깥"에서 조각난 빈 공간이 문제라서 외부 단편화다.

   > 각각을 줄이기 위한 전략을 서술하시오.

   내부 단편화 줄이기:
   ㄴ 정렬 단위를 줄이면 패딩이 줄어든다 (하지만 성능 트레이드오프)
   ㄴ 요청 크기에 딱 맞는 size class를 쓴다 (Segregated List)
   ㄴ 작은 블록 전용 allocator 사용 (slab allocator 등)

   외부 단편화 줄이기:
   ㄴ Coalescing: 인접한 빈 블록을 합쳐서 큰 블록으로 만든다
   ㄴ Best Fit: 요청 크기에 가장 가까운 빈 블록을 선택해서 잔여 조각을 최소화
   ㄴ Compaction: 사용 중인 블록을 한쪽으로 모아서 빈 공간을 연속으로 만든다
      (GC가 있는 언어에서 가능, C의 malloc에서는 포인터가 바뀌므로 불가능)

   > [검증] 힙에 [16B할당][48B가용][16B할당][32B가용] 상태에서 malloc(60)이 실패하는 이유를 내부/외부 단편화로 설명하시오.

   ```text
   ━━━ 현재 힙 상태 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

     [16B 할당][48B 가용][16B 할당][32B 가용]

     총 가용 공간 = 48 + 32 = 80B
     요청 크기 = 60B

   ━━━ 왜 실패하는가 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

     80B가 비어있지만 연속된 60B 블록이 없다.
     |
     ㄴ 48B 블록 → 60B 부족 (12B 모자람)
     ㄴ 32B 블록 → 60B 부족 (28B 모자람)
     ㄴ 두 블록 사이에 16B 할당 블록이 끼어있어서 합칠 수 없음

     이것이 외부 단편화다.
     빈 공간의 총합은 충분하지만 조각나 있어서 할당 불가능.

   ━━━ 만약 내부 단편화도 있다면 ━━━━━━━━━━━━━━━━━

     실제로는 헤더(8B) + 정렬 패딩까지 포함하면
     malloc(60) → 필요한 실제 블록 크기 = 60 + 8(헤더) = 68B
     정렬로 올림하면 72B 또는 80B가 될 수 있다.

     48B 가용 블록은 72B 요청에 더더욱 부족하다.

   ━━━ 해결하려면 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

     ㄴ 16B 할당 블록이 free되면 → coalesce로 48+16+32 = 96B 연속 확보
     ㄴ 애초에 배치 정책을 잘 했다면 조각이 덜 났을 수 있음
   ```

   </details>

2. [Q20. Segregated Free List — 왜 bin을 나누는가?](../q20-segregated-free-list.md)
   allocator가 성능과 단편화를 동시에 잡기 위해 어떤 구조를 쓰는지 설명한다.
   - Implicit → Explicit → Segregated Free List로 발전한 이유를 시간 복잡도 관점에서 설명하시오.
   - Segregated list에서 size class(bin) 설계 기준을 서술하시오.
   - [검증] bin이 {1-8B}, {9-16B}, {17-32B}일 때, malloc(12) 요청이 처리되는 과정을 설명하시오.

   <details>
   <summary>최우녕 답변</summary>

   > Implicit → Explicit → Segregated Free List로 발전한 이유를 시간 복잡도 관점에서 설명하시오.

   Implicit Free List:
   모든 블록(할당 + 가용)을 순서대로 탐색한다. 할당된 블록도 건너뛰면서 봐야 하므로
   malloc 호출 시 O(전체 블록 수)가 걸린다. 블록이 많아지면 매우 느리다.

   Explicit Free List:
   가용 블록만 연결 리스트로 묶는다. 할당된 블록은 건너뛴다.
   malloc 시 O(가용 블록 수)만 탐색하면 된다.
   하지만 가용 블록이 많으면 여전히 느릴 수 있다.

   Segregated Free List:
   가용 블록을 크기별로 분류한 여러 개의 리스트(bin)로 나눈다.
   malloc(n) 요청이 오면 n에 맞는 bin만 보면 되므로
   탐색 범위가 크게 줄어든다. 적절한 bin 설계 시 거의 O(1)에 가까워진다.

   ```text
     발전 방향:
     ㄴ Implicit : 전체 블록 순회      O(전체 블록 수)
     ㄴ Explicit : 가용 블록만 순회    O(가용 블록 수)
     ㄴ Segregated: 해당 bin만 탐색    ~O(1)
   ```

   > Segregated list에서 size class(bin) 설계 기준을 서술하시오.

   작은 크기는 정확하게, 큰 크기는 범위를 넓게 잡는다.
   작은 할당이 빈번하므로 정확한 크기로 bin을 만들면 낭비가 적고,
   큰 할당은 드물므로 넓은 범위를 하나의 bin으로 묶어도 괜찮다.

   일반적인 설계:
   ㄴ 작은 크기: 8B 단위로 정확히 (8, 16, 24, 32, ...)
   ㄴ 중간 크기: 2의 거듭제곱 범위 (33~64, 65~128, 129~256, ...)
   ㄴ 큰 크기: 하나의 큰 bin에 몰아넣기

   glibc malloc(ptmalloc)은 fastbin(작은 크기, LIFO), smallbin, largebin으로
   3단계로 나눠서 관리한다.

   > [검증] bin이 {1-8B}, {9-16B}, {17-32B}일 때, malloc(12) 요청이 처리되는 과정을 설명하시오.

   ```text
   전제: bin 구성
     bin[0]: {1-8B}   가용 블록들
     bin[1]: {9-16B}  가용 블록들
     bin[2]: {17-32B} 가용 블록들

   ━━━ malloc(12) 처리 과정 ━━━━━━━━━━━━━━━━━━━━━━━

     1. 요청 크기 결정
        |
        ㄴ 사용자 요청 = 12B
        ㄴ 헤더(8B) + 정렬 패딩 고려 → 실제 필요한 블록 크기 = 16B~24B
           (구현마다 다르지만, 여기선 요청 크기 12B 기준으로 bin 선택)

     2. bin 선택
        |
        ㄴ 12B → {9-16B} bin에 해당 → bin[1] 탐색

     3. bin[1]에서 블록 찾기
        |
        ㄴ 가용 블록 있음 → 그 블록을 할당, 리스트에서 제거
        ㄴ 가용 블록 없음 → bin[2] (더 큰 bin)에서 블록을 가져와 split
           |
           ㄴ bin[2]에서 32B 블록을 꺼냄
           ㄴ 16B로 split → 16B 할당, 나머지 16B는 bin[1]에 삽입
           |
           ㄴ bin[2]도 비어있으면 → sbrk/mmap으로 OS에 새 메모리 요청

     4. 할당 완료
        |
        ㄴ 16B 블록의 헤더에 할당 비트 설정
        ㄴ 헤더 다음 주소를 사용자에게 반환

   ━━━ 내부 단편화 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

     요청 12B인데 16B 블록을 할당 → 4B 패딩 낭비
     이것이 내부 단편화다. bin을 더 세밀하게 나누면 줄일 수 있지만
     bin 수가 많아지면 관리 오버헤드가 커진다.
   ```

   </details>

3. [Q21. 헤더의 하위 3비트 트릭 — 정렬이 만든 공짜 비트](../q21-header-lower-3bits.md)
   block metadata를 어떻게 압축해서 저장하는지 설명한다. allocator 구현 디테일의 핵심이다.
   - 블록 크기가 항상 8의 배수인 이유를 설명하시오.
   - 하위 3비트에 저장되는 플래그들을 나열하고 각각의 역할을 설명하시오.
   - [검증] 헤더값이 `0x00000019`일 때 블록 크기는? 할당 상태는? 다음 블록 헤더의 주소를 계산하시오.

   <details>
   <summary>최우녕 답변</summary>

   > 블록 크기가 항상 8의 배수인 이유를 설명하시오.

   x86-64에서 가장 큰 기본 타입은 double과 포인터로 둘 다 8바이트다.
   CPU가 데이터를 읽을 때 주소가 자연 정렬(natural alignment)되어 있으면
   한 번의 메모리 접근으로 읽을 수 있지만,
   정렬이 안 되면 두 번 접근해야 하거나 아예 에러가 난다.

   malloc이 반환하는 주소가 항상 8바이트 정렬이려면
   블록 시작 주소가 8의 배수여야 한다.
   블록 크기도 8의 배수로 만들면 다음 블록도 자동으로 8바이트 정렬된다.

   블록 크기가 8의 배수라는 건 이진수로 하위 3비트가 항상 000이라는 뜻이다.
   이 3비트는 크기 정보에 사용되지 않으므로 "공짜 비트"로 다른 용도로 쓸 수 있다.

   > 하위 3비트에 저장되는 플래그들을 나열하고 각각의 역할을 설명하시오.

   ```text
     헤더 (4바이트 or 8바이트):
     [  블록 크기 (상위 비트)  |  P  |  -  |  A  ]
                                bit2  bit1  bit0

     bit 0 (A): 현재 블록의 할당 상태
     ㄴ 1 = 할당됨 (allocated)
     ㄴ 0 = 가용 (free)

     bit 1: 구현에 따라 다름 (예약 비트)
     ㄴ glibc에서는 mmap으로 할당된 블록인지 표시

     bit 2 (P): 이전 블록의 할당 상태
     ㄴ 1 = 이전 블록이 할당됨
     ㄴ 0 = 이전 블록이 가용
     ㄴ coalesce 시 이전 블록을 합칠 수 있는지 빠르게 판단하기 위함
   ```

   블록 크기를 읽으려면 하위 3비트를 마스킹해서 제거한다:
   `size = header & ~0x7`   (하위 3비트를 0으로 만듦)

   할당 비트를 읽으려면:
   `is_alloc = header & 0x1`   (bit 0만 꺼냄)

   > [검증] 헤더값이 `0x00000019`일 때 블록 크기는? 할당 상태는? 다음 블록 헤더의 주소를 계산하시오.

   ```text
   ━━━ 헤더 분석 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

     헤더 = 0x19 = 0b 0001 1001

     하위 3비트: 001
     ㄴ bit 0 (A) = 1 → 현재 블록: 할당됨
     ㄴ bit 1     = 0
     ㄴ bit 2 (P) = 0 → 이전 블록: 가용

     블록 크기 = 0x19 & ~0x7
               = 0x19 & 0xFFFFFFF8
               = 0x18
               = 24바이트

     확인: 0x19 = 25 → 25 - 1(하위 비트) = 24 = 8 × 3 ✓

   ━━━ 다음 블록 헤더 주소 ━━━━━━━━━━━━━━━━━━━━━━━━

     현재 블록 헤더 주소를 hp라 하면:

     다음 블록 헤더 = hp + 블록 크기
                    = hp + 24

     예: hp = 0x1000이면
     ㄴ 현재 블록: 0x1000 ~ 0x1017 (24바이트)
     ㄴ 다음 블록 헤더: 0x1018

   ━━━ 정리 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

     헤더 0x19 → 블록 크기 24B, 현재 할당됨, 이전 블록 가용
     다음 블록 = 헤더 주소 + 24

     이 트릭 덕분에 헤더 하나에 크기와 상태 정보를 모두 담을 수 있다.
     별도의 상태 필드가 필요 없으니 메모리를 절약한다.
   ```

   </details>

4. [Q22. Split과 Coalesce — 경계 태그의 O(1) 마법](../q22-split-coalesce.md)
   free block을 어떻게 쪼개고 다시 합치는지 설명한다. allocator의 동작 원리 자체를 보여 준다.
   - split이 필요한 조건을 설명하시오.
   - free() 시 coalesce의 4가지 케이스를 설명하시오.
   - [검증] 풋터 없이 이전 블록 크기를 O(1)로 아는 방법을 설명하시오. (prev-allocated bit)

   <details>
   <summary>최우녕 답변</summary>

   > split이 필요한 조건을 설명하시오.

   malloc 요청이 들어왔을 때 찾은 가용 블록이 요청 크기보다 충분히 크면 split한다.
   "충분히 크다"는 건 쪼갠 나머지가 최소 블록 크기 이상이라는 뜻이다.

   최소 블록 크기는 보통 헤더(8B) + 최소 페이로드(8B) = 16B다.
   (Explicit Free List라면 prev/next 포인터 공간도 필요하므로 더 클 수 있다)

   ```text
     split 조건:
     가용 블록 크기 - 요청 블록 크기 >= 최소 블록 크기

     예: 가용 블록 = 64B, 요청 = 24B
     ㄴ 나머지 = 64 - 24 = 40B >= 16B(최소) → split 한다
     ㄴ 앞쪽 24B: 할당 블록으로 사용
     ㄴ 뒤쪽 40B: 새로운 가용 블록 (헤더 설정 후 free list에 삽입)

     예: 가용 블록 = 32B, 요청 = 24B
     ㄴ 나머지 = 32 - 24 = 8B < 16B(최소) → split 안 함
     ㄴ 32B 전체를 할당 → 8B 내부 단편화 발생
   ```

   split을 안 하면 내부 단편화, split을 하면 외부 단편화 가능성.
   이 트레이드오프가 allocator 설계의 핵심이다.

   > free() 시 coalesce의 4가지 케이스를 설명하시오.

   free()가 호출되면 현재 블록을 가용 상태로 바꾸고,
   인접한 블록이 이미 가용이면 합쳐서(coalesce) 큰 블록으로 만든다.
   이전/다음 블록의 할당 상태에 따라 4가지 케이스가 나온다.

   ```text
   ━━━ 전제: 현재 블록 = B, 이전 = PREV, 다음 = NEXT ━━━

     Case 1: PREV 할당, NEXT 할당
     |
     ㄴ 합칠 상대가 없다
     ㄴ B만 가용으로 표시하고 free list에 삽입
     ㄴ [PREV 할당][B 가용][NEXT 할당]

     Case 2: PREV 할당, NEXT 가용
     |
     ㄴ B와 NEXT를 합친다
     ㄴ B의 헤더에 합친 크기를 기록 (B 크기 + NEXT 크기)
     ㄴ NEXT를 free list에서 제거, 합친 블록을 삽입
     ㄴ [PREV 할당][B+NEXT 가용]

     Case 3: PREV 가용, NEXT 할당
     |
     ㄴ PREV와 B를 합친다
     ㄴ PREV의 헤더에 합친 크기를 기록 (PREV 크기 + B 크기)
     ㄴ PREV는 이미 free list에 있으므로 크기만 갱신
     ㄴ [PREV+B 가용][NEXT 할당]

     Case 4: PREV 가용, NEXT 가용
     |
     ㄴ PREV + B + NEXT 세 블록을 모두 합친다
     ㄴ PREV 헤더에 합친 크기 기록 (PREV + B + NEXT)
     ㄴ NEXT를 free list에서 제거, PREV 크기 갱신
     ㄴ [PREV+B+NEXT 가용]

   ━━━ 핵심 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

     다음 블록 상태: 다음 블록 헤더의 할당 비트를 읽으면 된다 → O(1)
     이전 블록 상태: 현재 블록 헤더의 P비트(bit 2)를 읽으면 된다 → O(1)
     이전 블록 크기: 풋터(경계 태그)에서 읽는다 → O(1)

     4가지 케이스 모두 O(1)에 처리된다.
     이것이 "경계 태그의 O(1) 마법"이다.
   ```

   > [검증] 풋터 없이 이전 블록 크기를 O(1)로 아는 방법을 설명하시오. (prev-allocated bit)

   기본적으로 이전 블록의 크기를 알려면 이전 블록의 풋터(boundary tag)를 읽는다.
   풋터는 블록 끝에 헤더를 복사한 것이다. 현재 블록 헤더 바로 앞 워드가 이전 블록의 풋터다.

   그런데 풋터는 가용 블록에만 필요하다. 왜?
   coalesce는 이전 블록이 "가용"일 때만 합치니까,
   이전 블록이 "할당"이면 크기를 알 필요가 없다.

   ```text
   ━━━ 최적화: 할당 블록의 풋터 제거 ━━━━━━━━━━━━━━━━

     현재 블록 헤더의 P비트 (bit 2) = prev-allocated bit

     ㄴ P = 1 (이전 블록 할당됨):
       합칠 필요 없음 → 이전 블록 크기를 몰라도 됨
       이전 블록의 풋터가 필요 없다 → 그 공간을 페이로드로 쓸 수 있다

     ㄴ P = 0 (이전 블록 가용):
       합쳐야 할 수 있음 → 이전 블록 크기가 필요
       이전 블록은 가용이므로 풋터가 있다 → 풋터에서 크기를 읽는다

   ━━━ 정리 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

     규칙:
     ㄴ 가용 블록 → 풋터를 유지한다 (coalesce에 필요)
     ㄴ 할당 블록 → 풋터를 제거한다 (P비트만으로 충분)

     이 최적화로 할당 블록마다 4~8바이트를 절약한다.
     블록이 수천 개면 절약량이 상당하다.

     glibc ptmalloc이 실제로 이 방식을 사용한다.
     CSAPP에서 "최적화된 경계 태그"로 소개하는 기법이다.
   ```

5. [Q23. 배치 정책 — First/Next/Best Fit 실전 트레이드오프](../q23-placement-policy.md)
   어떤 free block을 선택할지가 실제 성능과 단편화에 어떤 차이를 만드는지 정리한다.
   - 세 가지 배치 정책(First Fit, Next Fit, Best Fit)의 시간 복잡도와 단편화 경향을 비교하시오.
   - [검증] 가용리스트 [32B, 64B, 24B, 128B]에서 malloc(20)(=24B 필요) 시 각 정책이 선택하는 블록과 split 잔여 크기를 구하시오.

   <details>
   <summary>최우녕 답변</summary>

   > 세 가지 배치 정책(First Fit, Next Fit, Best Fit)의 시간 복잡도와 단편화 경향을 비교하시오.

   ```text
   ━━━ First Fit ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

     방법: free list의 처음부터 탐색해서 처음 맞는 블록을 선택
     시간: O(가용 블록 수) — 최악의 경우 전체 리스트 순회
     단편화:
     ㄴ 리스트 앞쪽에 작은 조각이 쌓이는 경향 (앞쪽을 계속 쪼개니까)
     ㄴ 뒤쪽에는 큰 블록이 남아있을 가능성
     ㄴ 시간이 지나면 앞쪽 탐색이 느려진다

   ━━━ Next Fit ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

     방법: 마지막으로 탐색을 멈춘 위치부터 이어서 탐색
     시간: O(가용 블록 수) — 최악은 같지만 평균적으로 First Fit보다 빠르다
     단편화:
     ㄴ 조각이 리스트 전체에 고르게 퍼진다
     ㄴ 큰 블록도 쪼개지기 쉬워서 First Fit보다 외부 단편화가 더 심할 수 있다
     ㄴ 연구 결과 실전에서 First Fit보다 메모리 이용률이 낮다

   ━━━ Best Fit ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

     방법: 전체 free list를 탐색해서 요청 크기에 가장 가까운 블록을 선택
     시간: O(가용 블록 수) — 항상 전체 순회 (단, segregated list에서는 ~O(1))
     단편화:
     ㄴ split 후 잔여 조각이 가장 작다 → 내부 단편화 최소
     ㄴ 하지만 아주 작은 조각이 많이 생김 → 쓸모없는 외부 단편화 발생 가능
     ㄴ 종합적으로 메모리 이용률은 가장 좋은 편

   ━━━ 비교 요약 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

     정책       | 탐색 속도  | 단편화     | 실전 평가
     -----------|-----------|-----------|------------------
     First Fit  | 중간      | 앞쪽 집중  | 단순하고 무난
     Next Fit   | 빠름      | 고르게 나쁨 | 큰 블록 보존 못함
     Best Fit   | 느림      | 가장 적음  | segregated list와 결합 시 최적
   ```

   > [검증] 가용리스트 [32B, 64B, 24B, 128B]에서 malloc(20)(=24B 필요) 시 각 정책이 선택하는 블록과 split 잔여 크기를 구하시오.

   ```text
     전제: 최소 블록 크기 = 16B (헤더 8B + 최소 페이로드 8B)
     요청: malloc(20) → 정렬 후 필요한 블록 크기 = 24B

     가용리스트 순서: [32B] → [64B] → [24B] → [128B]

   ━━━ First Fit ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

     |
   ```

6. [Q24. free()의 설계 철학 — 왜 에러를 알려주지 않는가?](../q24-free-design.md)
   allocator API가 왜 위험하지만 그렇게 설계되었는지를 설명한다. 개발자 관점의 현실 문제와 연결된다.
   - C 표준이 free(invalid)를 UB로 정의한 이유를 설명하시오.
   - 잘못된 포인터를 free 했을 때 일어날 수 있는 일을 3가지 이상 서술하시오.
   - [검증] 다음 각 경우에 어떤 문제가 발생하는지 설명하시오: (a) double free (b) free(p+1) (c) int x; free(&x)

   <details>
   <summary>최우녕 답변</summary>

   > C 표준이 free(invalid)를 UB로 정의한 이유를 설명하시오.

   free()는 malloc이 반환한 포인터만 받도록 설계되어 있다.
   잘못된 포인터를 검증하려면 free가 호출될 때마다
   "이 주소가 정말 malloc으로 할당된 블록의 시작 주소인가"를 확인해야 하는데,
   이걸 매번 하면 성능이 크게 떨어진다.

   C의 설계 철학은 "프로그래머를 신뢰하고, 런타임 오버헤드를 최소화한다"이다.
   그래서 잘못된 포인터로 free를 호출하면 에러를 알려주는 대신
   Undefined Behavior(정의되지 않은 동작)로 남겨뒀다.
   검증 비용을 프로그래머의 책임으로 넘긴 것이다.

   free()의 반환값이 void인 것도 같은 이유다.
   성공/실패를 알려주려면 매번 검증을 해야 하는데, C는 그 비용을 거부한 거다.

   > 잘못된 포인터를 free 했을 때 일어날 수 있는 일을 3가지 이상 서술하시오.

   1. 힙 메타데이터 파괴: free는 포인터 앞에 있는 헤더(블록 크기, 할당 비트)를 읽는다.
      잘못된 주소의 헤더를 읽으면 엉뚱한 메모리를 해제하거나 free list가 깨진다.

   2. 다른 블록 침범: 깨진 free list 때문에 다음 malloc이 이미 사용 중인 블록을
      반환할 수 있다. 두 포인터가 같은 메모리를 가리키게 되어 데이터가 덮어씌워진다.

   3. Segmentation Fault: 잘못된 주소가 프로세스에 할당되지 않은 VA 영역이면
      MMU가 페이지 테이블을 뒤져도 해당 PTE가 invalid(커널이 할당한 적 없음)이다.
      MMU는 "이 주소 변환 못 함"이라고 CPU에 예외를 보내고,
      커널 폴트 핸들러가 "복구 불가능한 접근"이라고 판단하면
      프로세스에 SIGSEGV(Segmentation Fault) 시그널을 보내서 강제 종료한다.
      q04의 Invalid 상태(PTE 3번째 상태)와 같은 경우다.

   4. 보안 취약점: 공격자가 힙 메타데이터를 의도적으로 조작하면
      임의 주소에 쓰기(arbitrary write)가 가능해져서 코드 실행 공격으로 이어질 수 있다.

   > [검증] 다음 각 경우에 어떤 문제가 발생하는지 설명하시오: (a) double free (b) free(p+1) (c) int x; free(&x)

   ```text
   ━━━ 먼저: malloc 블록의 구조 이해 ━━━━━━━━━━━━━━

     malloc(100) 하면 실제 메모리는 이렇게 생겼다:

     [ 헤더 (8B) ][ 사용자 데이터 (100B) ]
          ↑              ↑
      블록 크기/할당비트    malloc이 반환한 포인터 p

     free(p) 하면 free는 p 바로 앞 8바이트를 헤더로 읽어서
     "이 블록이 몇 바이트짜리구나"를 알아내고 free list에 넣는다.
     이 구조를 모르면 아래 3가지가 왜 위험한지 이해가 안 된다.

   ━━━ (a) double free ━━━━━━━━━━━━━━━━━━━━━━━━━━━━

     char *p = malloc(100);
     free(p);     // 1. 블록을 free list에 넣음
     free(p);     // 2. 같은 블록을 또 넣음

     free list는 "여기 빈 블록 있어요" 목록이다.
     같은 블록이 두 번 등록되면:

     ㄴ 다음에 malloc을 두 번 호출하면 같은 주소를 두 번 반환함
     ㄴ char *a = malloc(100);  // p의 주소 반환
     ㄴ char *b = malloc(100);  // 또 p의 주소 반환
     ㄴ a와 b가 같은 메모리를 가리킴
     ㄴ a에 쓰면 b의 데이터가 덮어씌워짐 → 원인 찾기 매우 어려운 버그

     방지: free 후 반드시 p = NULL; 하면 두 번째 free(NULL)은 아무것도 안 함

   ━━━ (b) free(p+1) ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

     char *p = malloc(100);
     free(p + 1);  // 블록 시작이 아닌 1바이트 뒤 주소

     free는 전달받은 주소 바로 앞 8바이트를 헤더로 읽는다.
     근데 p+1 앞 8바이트에는 헤더가 아니라 사용자 데이터가 있다.

     [ 헤더 ][ 데이터 ... ]
              ↑ p  ↑ p+1
                    free가 여기 앞 8B를 헤더로 착각

     ㄴ 엉뚱한 값을 "블록 크기 = 3억 바이트"처럼 해석할 수 있음
     ㄴ 힙의 free list 구조 전체가 깨짐
     ㄴ 이후 malloc/free 모두 비정상 동작
     ㄴ 바로 크래시하면 차라리 낫고, 조용히 데이터가 오염되면 더 위험

   ━━━ (c) int x; free(&x) ━━━━━━━━━━━━━━━━━━━━━━━

     int x = 42;
     free(&x);  // 스택 변수의 주소를 free

     메모리 영역은 크게 스택, 힙, 코드, 데이터로 나뉜다.
     malloc은 힙 영역만 관리한다. &x는 스택 영역의 주소다.

     free는 &x 앞 8바이트를 헤더로 읽으려 하지만
     스택에는 malloc 헤더가 없다. 거기엔 다른 지역 변수나
     함수 리턴 주소가 있다.

     ㄴ 스택 메모리를 힙 블록으로 착각하고 free list에 등록
     ㄴ 다음 malloc이 스택 영역을 "빈 힙 블록"으로 반환할 수 있음
     ㄴ 스택과 힙이 뒤섞이면 함수 리턴 주소가 덮어씌워져서 크래시
     ㄴ 공격자가 악용하면 원하는 코드를 실행할 수도 있음

   ━━━ 공통 교훈 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

     free()는 "이 포인터가 올바른가"를 검증하지 않는다.
     전달받은 주소 앞을 무조건 헤더로 읽기 때문에
     잘못된 주소를 넘기면 힙 전체가 오염될 수 있다.

     프로그래머가 지켜야 할 규칙:
     ㄴ malloc이 반환한 원래 주소로만 free
     ㄴ free 후 포인터를 NULL로 설정
     ㄴ 스택/전역 변수 주소를 절대 free하지 않기
   ```

   </details>

7. [Q25. 메모리 버그 총정리 — 왜 C가 위험한가?](../q25-memory-bugs.md)
   마지막 정리 질문이다. 지금까지 다룬 구조가 왜 leak, double free, use-after-free, overflow 같은 버그로 이어지는지를 묶어 준다.
   - 대표적인 메모리 버그 5가지를 코드 예시와 함께 설명하시오.
   - dangling pointer와 memory leak의 차이를 설명하시오.
   - [검증] `int **A = malloc(n * sizeof(int)); ... A[n] = NULL;` 에서 버그 2개를 찾고 수정하시오.

   <details>
   <summary>최우녕 답변</summary>

   > 대표적인 메모리 버그 5가지를 코드 예시와 함께 설명하시오.

   ```text
   ━━━ 1. Buffer Overflow (버퍼 오버플로우) ━━━━━━━━━━

     char buf[8];
     strcpy(buf, "Hello, World!");  // 13B를 8B 버퍼에 복사

     ㄴ buf 뒤에 있는 메모리(다른 변수, 리턴 주소 등)가 덮어씌워진다
     ㄴ 힙이면 다음 블록의 헤더가 파괴 → allocator 오동작
     ㄴ 스택이면 리턴 주소 변조 → 공격자가 코드 실행 가능

   ━━━ 2. Use-After-Free (해제 후 사용) ━━━━━━━━━━━━━

     char *p = malloc(100);
     free(p);
     p[0] = 'A';  // 이미 해제된 메모리에 쓰기

     ㄴ free 후 그 블록은 free list에 들어가고 다른 malloc이 재사용할 수 있다
     ㄴ p로 쓰면 새로 할당된 다른 데이터를 오염시킨다
     ㄴ 읽기도 위험 — 쓰레기 값이거나 다른 사용자의 데이터일 수 있음

   ━━━ 3. Double Free (이중 해제) ━━━━━━━━━━━━━━━━━━━

     char *p = malloc(100);
     free(p);
     free(p);  // 같은 블록을 두 번 해제

     ㄴ free list에 같은 블록이 두 번 등록됨
     ㄴ 이후 malloc 두 번이 같은 주소를 반환 → 두 포인터가 같은 메모리 공유
     ㄴ Q24에서 상세히 다룸

   ━━━ 4. Memory Leak (메모리 누수) ━━━━━━━━━━━━━━━━━

     void foo() {
         char *p = malloc(100);
         return;  // free 안 하고 함수 종료
     }

     ㄴ p는 스택에 있던 지역 변수 → 함수 종료 시 사라짐
     ㄴ malloc한 힙 메모리는 여전히 할당 상태 → 접근할 방법도, 해제할 방법도 없음
     ㄴ 장시간 실행되는 서버에서 leak이 쌓이면 OOM(Out of Memory)으로 크래시

   ━━━ 5. Uninitialized Read (초기화 안 된 메모리 읽기) ━

     int *p = malloc(sizeof(int));
     printf("%d\n", *p);  // 초기화 안 하고 읽기

     ㄴ malloc은 메모리를 0으로 초기화하지 않는다 (calloc은 함)
     ㄴ 이전에 그 주소를 썼던 프로그램의 잔여 데이터가 남아있다
     ㄴ 비결정적 동작 — 디버깅 시 재현이 안 될 수 있음
     ㄴ 보안: 이전 사용자의 비밀번호, 키 등이 노출될 가능성
   ```

   > dangling pointer와 memory leak의 차이를 설명하시오.

   ```text
   ━━━ Dangling Pointer (허상 포인터) ━━━━━━━━━━━━━━━

     포인터는 남아있는데 가리키는 메모리가 해제된 상태.
     "열쇠는 있는데 문이 철거된 것"

     char *p = malloc(100);
     free(p);
     // p는 여전히 이전 주소를 가리킴 → dangling pointer
     // p를 통해 읽기/쓰기하면 use-after-free 버그

   ━━━ Memory Leak (메모리 누수) ━━━━━━━━━━━━━━━━━━━━

     메모리는 할당되어 있는데 가리키는 포인터가 없는 상태.
     "문은 있는데 열쇠를 잃어버린 것"

     char *p = malloc(100);
     p = malloc(200);  // 이전 100B의 주소를 잃어버림
     // 첫 번째 100B는 할당 상태이지만 접근 불가 → leak

   ━━━ 핵심 차이 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

     Dangling Pointer: 포인터 O, 메모리 X → 접근하면 위험
     Memory Leak:      포인터 X, 메모리 O → 해제 불가, 자원 낭비

     둘 다 포인터와 메모리의 수명이 불일치할 때 발생한다.
     C에서는 프로그래머가 이 수명을 직접 관리해야 하므로
     두 버그 모두 자연스럽게 발생하기 쉽다.
   ```

   > [검증] `int **A = malloc(n * sizeof(int)); ... A[n] = NULL;` 에서 버그 2개를 찾고 수정하시오.

   ```text
     원본 코드:
     int **A = malloc(n * sizeof(int));
     // ... 사용 ...
     A[n] = NULL;

   ━━━ 버그 1: sizeof 타입 불일치 ━━━━━━━━━━━━━━━━━━

     A는 int** (포인터의 포인터)다.
     A의 각 원소는 int* (포인터)다.

     malloc(n * sizeof(int)) → int 크기(4B)로 n개 할당
     실제 필요한 건 int* 크기(8B)로 n개 할당

     x86-64에서:
     ㄴ sizeof(int) = 4바이트
     ㄴ sizeof(int*) = 8바이트
     ㄴ 절반만 할당됨 → 뒤쪽 원소에 접근하면 버퍼 오버플로우

     수정: malloc(n * sizeof(int*))

   ━━━ 버그 2: 배열 범위 초과 (off-by-one) ━━━━━━━━━

     A[n] = NULL;

     A는 n개 원소를 할당했으므로 인덱스는 0 ~ n-1까지 유효하다.
     A[n]은 할당 범위 바로 다음 메모리에 쓰기 → 버퍼 오버플로우

     ㄴ 다음 블록의 헤더를 NULL(0)로 덮어쓸 수 있음
     ㄴ allocator 메타데이터 파괴 → 이후 malloc/free 비정상 동작

     수정: A[n-1] = NULL;

   ━━━ 수정된 코드 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

     int **A = malloc(n * sizeof(int*));  // int → int*
     // ... 사용 ...
     A[n-1] = NULL;                       // A[n] → A[n-1]
   ```

   </details>

## 발표 마무리 문장

이 발표가 끝나면 청중은 "메모리 버그는 단순 부주의가 아니라, allocator 구조와 수동 메모리 관리 모델에서 자연스럽게 발생하는 위험"이라는 점을 이해하게 된다.
이 파트가 전체 발표의 현실적인 착지점이 된다.
