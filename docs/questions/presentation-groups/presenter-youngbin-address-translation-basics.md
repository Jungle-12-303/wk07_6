# 발표 1. 주소와 주소 번역의 기초

## 이 발표의 역할

이 파트는 "C 코드의 포인터가 어떻게 실제 메모리에 닿는가"를 가장 처음부터 설명하는 역할이다.
즉 VA와 PA의 차이, 메모리 계층의 단위, 비트 분해, 페이지 테이블, TLB, 다단계 페이지 테이블까지를 하나의 흐름으로 묶는다.

## 한 줄 흐름

`VA/PA` -> `page/cache line/word` -> `VPN + offset` -> `page table` -> `TLB` -> `multi-level page table`

## 담당 질문 순서

1. [Q1. VA와 PA의 정체 — '번역'이지 '저장'이 아니다](../q01-va-pa.md)
   시작점이다. 가상 메모리를 "가짜 메모리"가 아니라 "주소 변환 체계"로 잡아 주어야 뒤 질문들이 모두 연결된다.

2. [Q2. 페이지·캐시라인·워드 — 계층이 다르다](../q02-page-cacheline-word.md)
   CPU, cache, DRAM, disk가 서로 다른 단위를 쓴다는 점을 설명한다. 이후 page 단위 논의의 배경이 된다.

3. [Q10. VPN + Offset 분해 — 비트 연산으로 이해하기](../q10-vpn-offset.md)
   이제 주소를 추상 개념이 아니라 실제 비트 필드로 본다. Q11의 page table lookup으로 자연스럽게 이어진다.

4. [Q11. 페이지 테이블·PTE·주소 번역 전체 파이프라인](../q11-page-table-pipeline.md)
   VA가 실제로 PA가 되는 핵심 절차를 설명한다. 이 발표의 중심 질문이다.

5. [Q13. TLB — page-table walk의 숏컷](../q13-tlb.md)
   번역이 왜 느릴 수 있는지, 그래서 TLB가 왜 필요한지를 설명한다. 성능 관점이 여기서 들어온다.

6. [Q14. 다단계 페이지 테이블 — '안 쓰는 건 안 만든다'](../q14-multi-level-page-table.md)
   단일 page table이 왜 비현실적인지와 sparse address space를 어떻게 다루는지를 설명한다.

## 발표 마무리 문장

이 발표가 끝나면 청중은 "프로그램이 낸 주소가 어떤 자료구조와 하드웨어를 거쳐 실제 물리 주소가 되는가"를 설명할 수 있어야 한다.
다음 발표는 이 번역 체계가 실제 실행 중에 어떤 fault, protection, residency 문제를 만드는지로 넘어가면 자연스럽다.
