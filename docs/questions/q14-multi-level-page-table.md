# Q14. 다단계 페이지 테이블 — '안 쓰는 건 안 만든다'

> CSAPP 9.6 | 5부. 다단계·TLB | 검증

## 질문

1. 단일 레벨 페이지 테이블의 메모리 낭비 문제를 계산과 함께 설명하시오.
2. 다단계 페이지 테이블이 메모리를 절약하는 원리를 서술하시오.
3. [검증] 2-level 페이지 테이블에서 VA → PA 변환 과정을 VPN1, VPN2, VPO를 사용하여 단계별로 설명하시오.

## 답변

### 최우녕

> 단일 레벨 페이지 테이블의 메모리 낭비 문제를 계산과 함께 설명하시오.

단일 페이지 테이블은 VPN을 인덱스로 하는 하나의 큰 배열이다.
x86-64 기준 VA 48비트, 4KB 페이지이면 VPN은 36비트.

엔트리 수 = 2^36 = 약 687억 개
PTE 1개 = 8바이트
테이블 크기 = 2^36 × 8B = 512GB

배열이기 때문에 실제로 사용하는 페이지가 몇 개든
687억 칸 전부가 메모리에 존재해야 한다. 빈칸을 빼면 인덱스로 접근할 수 없으니까.
프로세스 하나당 512GB는 DRAM에 담을 수 없다.

> 다단계 페이지 테이블이 메모리를 절약하는 원리를 서술하시오.

36비트 VPN을 한 번에 보지 않고 9비트씩 4단계로 나눈다.
각 단계 테이블은 2^9 = 512개 엔트리 = 4KB = 물리 페이지 1장이다.

절약의 핵심은 "안 쓰는 VA 영역의 테이블을 아예 안 만든다"는 것이다.
PML4에서 NULL인 엔트리는 그 아래 PDPT, PD, PT를 전부 생략한다.
PML4의 엔트리 하나가 512GB 영역을 커버하는데,
NULL 하나로 512GB 분량의 하위 테이블을 통째로 건너뛴다.

단일 테이블은 빈칸도 전부 만들어야 하는 배열이지만,
다단계 테이블은 사용하는 영역만 만드는 트리 구조다.

참고 — 64비트인데 왜 VA는 48비트만 쓰는가:
x86-64의 레지스터는 64비트(8바이트)라서 이론상 2^64 = 16EB 주소 공간이 가능하다.
하지만 16EB는 현재 기술로 절대 채울 수 없는 양이다.
64비트 주소를 전부 지원하려면 페이지 테이블 단계가 더 많아지고,
주소 버스 핀 수, TLB 태그 비트도 커진다. 다 비용이다.
그래서 Intel이 "당분간 48비트(256TB)면 충분하다"고 설계한 것이다.
64비트 레지스터에 주소를 넣을 때 상위 16비트는 47번 비트의 부호 확장으로 채운다.
나중에 메모리가 256TB를 넘기면 확장할 수 있도록 5-Level Paging(57비트, 128PB)도 이미 만들어져 있다.

참고 — 왜 하필 9비트씩 나누는가:
9비트 = 2^9 = 512개 엔트리. PTE 하나가 8바이트이므로
512 × 8 = 4096 = 4KB = 물리 페이지 딱 1장이다.
테이블 하나가 페이지 1장에 깔끔하게 들어가도록 9비트로 나눈 것이다.
9 × 4단계 = 36비트, VPN 전체를 정확히 소화한다.

참고 — PTE(8바이트) 안에 뭐가 들어있는가:

```text
PTE (64비트 = 8바이트):

  bit 63   bit 51          bit 12   bit 11    bit 0
  [ NX  |  PPN 영역 (40비트) |  플래그 (12비트)  ]

  1~3단계 PTE: 다음 단계 테이블의 물리 주소가 들어있음
  4단계(PT) PTE: 최종 PPN이 들어있음

  주요 플래그:
  ㄴ bit 0: Present (1 = 유효, 0 = 없음)
  ㄴ bit 1: Read/Write (0 = 읽기만, 1 = 읽기+쓰기)
  ㄴ bit 2: User/Supervisor (0 = 커널만, 1 = 유저도)
  ㄴ bit 5: Accessed (MMU가 접근하면 1로 세팅)
  ㄴ bit 6: Dirty (쓰기 발생 시 1로 세팅)
```

참고 — 물리 주소(PA) 비트 수는 CPU마다 다르다:
PTE의 PPN 영역은 40비트 자리가 있지만, 실제 CPU가 지원하는 PA 비트 수만 사용한다.

```text
  CPU 종류               PA 비트   최대 물리 메모리
  ─────────────────────  ────────  ────────────────
  일반 데스크탑           39비트    512GB
  서버 (Xeon 등)         46~52비트  64TB ~ 4PB
```

그릇(PTE)은 64비트로 통일해두고, CPU가 지원하는 만큼만 채워 쓴다.
나중에 물리 메모리가 커져도 PTE 형식을 바꿀 필요 없이 비트를 더 채우면 된다.

> [검증] 2-level 페이지 테이블에서 VA → PA 변환 과정을 VPN1, VPN2, VPO를 사용하여 단계별로 설명하시오.

```text
전제: 간단한 2-level 예시
      VA 16비트, 페이지 4KB(12비트 오프셋)은 너무 작으니
      VA 20비트, 페이지 1KB(10비트 오프셋)으로 가정
      VPN = 10비트 → 5비트씩 2단계

━━━ 1. VA 분해 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  VA 20비트:
  [  VPN1 (5bit)  |  VPN2 (5bit)  |  VPO (10bit)  ]
    1단계 인덱스     2단계 인덱스     페이지 내 오프셋

  예: VA = 0x72C00
  VPN1 = bits[19:15]
  VPN2 = bits[14:10]
  VPO  = bits[9:0]

━━━ 2. 변환 과정 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  CR3 (1단계 테이블 베이스 주소)
  |
  ㄴ 1단계 테이블[VPN1] 조회
     |
     ㄴ NULL → 이 영역은 할당 안 됨 → Segfault
     ㄴ 값 있음 → 2단계 테이블의 물리 베이스 주소
        |
        ㄴ 2단계 테이블[VPN2] 조회
           |
           ㄴ Invalid → Page Fault → 커널 처리
           ㄴ Valid → PPN 획득

  PA = (PPN << 10) | VPO

━━━ 3. x86-64는 4-level (같은 원리) ━━━━━━━━━━━━

  VPN 36비트를 9비트씩 4등분:

  [VPN1(9)][VPN2(9)][VPN3(9)][VPN4(9)][VPO(12)]
   PML4     PDPT     PD       PT       오프셋

  CR3 → PML4[VPN1] → PDPT[VPN2] → PD[VPN3] → PT[VPN4] → PPN

  2-level이든 4-level이든 원리는 동일하다.
  단계가 많을수록 더 큰 주소 공간을 커버하면서
  안 쓰는 영역을 더 세밀하게 생략할 수 있다.
```
