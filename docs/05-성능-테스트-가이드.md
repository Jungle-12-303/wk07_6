# 05. 성능 테스트 가이드 — INDEX_LOOKUP vs TABLE_SCAN

minidb의 B+ Tree 인덱스 효과를 직접 확인하는 테스트 시나리오이다.
동일한 1건을 검색할 때, 인덱스가 있는 칼럼(id)과 없는 칼럼(email)의 성능 차이를 비교한다.

---

## 1. 환경 준비

### 1-1. Docker 컨테이너 진입

```bash
docker exec -it <컨테이너ID> bash
cd /workspaces/SW_AI-W07-SQL
```

### 1-2. 빌드

```bash
make clean && make all
```

> ASan(AddressSanitizer)이 포함된 기본 빌드이다.
> 데이터 생성 시에는 ASan 없이 별도 빌드한다 (아래 참고).

---

## 2. 더미 데이터 생성

### 2-1. gen_data 빌드 (ASan 제외, 최적화 O2)

```bash
gcc -Wall -Wextra -O2 -g -Iinclude -DNDEBUG \
    -Wno-unused-result -Wno-stringop-truncation -Wno-maybe-uninitialized \
    -o build/gen_data_fast \
    tools/gen_data.c \
    src/storage/pager.c src/storage/schema.c src/storage/table.c \
    src/storage/bptree.c src/sql/parser.c src/sql/planner.c src/sql/executor.c \
    -lpthread
```

### 2-2. 데이터 생성

```bash
# 기존 DB 삭제 후 생성
rm -f sql.db

# 1,000만 건 생성 (약 30~60분 소요, ~850MB)
./build/gen_data_fast sql.db 10000000

# 빠르게 테스트하려면 100만 건 (약 1~2분, ~130MB)
./build/gen_data_fast sql.db 1000000
```

### 스키마

| 칼럼  | 타입         | 크기   | 인덱스 | 예시                     |
|-------|-------------|--------|--------|--------------------------|
| id    | BIGINT      | 8바이트 | B+ Tree | 500000                  |
| name  | VARCHAR(32) | 32바이트 | 없음   | Alice Kim               |
| email | VARCHAR(32) | 32바이트 | 없음   | user500000@test.com     |
| age   | INT         | 4바이트  | 없음   | 25                      |

행 크기: 76바이트 (8 + 32 + 32 + 4)

### 완료 확인

파일 크기로 대략적인 진행률을 확인할 수 있다.

```bash
ls -lh sql.db
```

| 행 수       | 예상 파일 크기 |
|-------------|---------------|
| 100만 건    | ~130MB        |
| 500만 건    | ~500MB        |
| 1,000만 건  | ~850MB        |

### 3-1. minidb 실행

## 4. 성능 비교 테스트

### 4-1. 디버그 모드 켜기

```
minidb> .debug
디버그 모드: ON
```

디버그 모드에서는 매 쿼리마다 다음 정보가 출력된다.

| 항목       | 의미                                        |
|-----------|---------------------------------------------|
| 소요       | 쿼리 실행 시간 (밀리초)                       |
| 페이지 로드 | pager_get_page() 호출 총 수                  |
| 히트       | 캐시에 이미 있어서 디스크 I/O 없이 사용한 횟수    |
| 미스       | 캐시에 없어서 디스크에서 새로 읽어온 횟수         |
| 디스크 기록 | dirty 프레임이 evict되어 디스크에 기록된 횟수     |

### 4-2. INDEX_LOOKUP — id로 검색 (B+ Tree, O(log n))

```
minidb> SELECT * FROM users WHERE id = 5000000
```

예상 출력:
```
id | name | email | age
----------+-----------+-----------+-----------
5000000 | Derek Choi | user5000000@test.com | 61
1행 조회 (INDEX_LOOKUP)
[debug] 소요: 0.02ms | 페이지 로드: 6 (히트: 1, 미스: 5) | 디스크 기록: 0
```

**핵심**: B+ Tree 높이가 4이면, root → internal → internal → leaf → heap = **5~6페이지만 읽는다**.

### 4-3. TABLE_SCAN — email로 검색 (풀스캔, O(n))

```
minidb> SELECT * FROM users WHERE email = 'user5000000@test.com'
```

예상 출력:
```
id | name | email | age
----------+-----------+-----------+-----------
5000000 | Derek Choi | user5000000@test.com | 61
1행 조회 (TABLE_SCAN)
[debug] 소요: 2800ms | 페이지 로드: 약 130000 (히트: 0, 미스: 약 130000) | 디스크 기록: 0
```

**핵심**: 인덱스가 없으므로 **모든 힙 페이지(~13만 페이지)를 순차 스캔**해야 한다.

### 4-4. 결과 비교

| 항목         | INDEX_LOOKUP (id)     | TABLE_SCAN (email)       |
|-------------|----------------------|--------------------------|
| 접근 경로    | B+ Tree 인덱스        | 힙 전체 스캔              |
| 시간 복잡도  | O(log n)             | O(n)                     |
| 페이지 로드  | 5~6                  | ~130,000                 |
| 소요 시간    | ~0.02ms              | ~2,800ms                 |
| 성능 차이    | —                    | **약 100,000배 느림**     |

같은 1건을 찾는데, 인덱스 유무로 이만큼의 차이가 발생한다.

## 5. 추가 명령어

### 실행 계획 확인 (EXPLAIN)

```
minidb> EXPLAIN SELECT * FROM users WHERE id = 5000000
Access Path: INDEX_LOOKUP
  Index: B+ Tree (id)
  Target: id = 5000000

minidb> EXPLAIN SELECT * FROM users WHERE email = 'user5000000@test.com'
Access Path: TABLE_SCAN
  Filter: email = 'user5000000@test.com'
  Scan: all heap pages
```

### 메타 명령어 목록

| 명령어   | 설명                                  |
|---------|---------------------------------------|
| .stats  | DB 통계 (행 수, 페이지 수, 트리 높이)    |
| .debug  | 쿼리 디버그 모드 ON/OFF 토글            |
| .pages  | 페이지 유형별 개수                      |
| .btree  | B+ Tree 구조 출력                      |
| .log    | pager flush 로그 ON/OFF               |
| .flush  | 모든 dirty 페이지 수동 디스크 기록       |
| .exit   | 종료                                  |

## 6. 전체 시나리오 (복사-붙여넣기용)

Docker 진입부터 테스트 완료까지 한 번에 실행할 수 있는 명령어 모음이다.

```bash
# ── 1. Docker 진입 ──
docker exec -it <컨테이너ID> bash
cd /workspaces/SW_AI-W07-SQL

# ── 2. 빌드 ──
make clean && make all

# ── 3. gen_data 빌드 (ASan 제외) ──
gcc -Wall -Wextra -O2 -g -Iinclude -DNDEBUG \
    -Wno-unused-result -Wno-stringop-truncation -Wno-maybe-uninitialized \
    -o build/gen_data_fast \
    tools/gen_data.c \
    src/storage/pager.c src/storage/schema.c src/storage/table.c \
    src/storage/bptree.c src/sql/parser.c src/sql/planner.c src/sql/executor.c \
    -lpthread

# ── 4. 더미 데이터 생성 (100만 건 = 빠른 테스트) ──
rm -f sql.db
./build/gen_data_fast sql.db 1000000

# ── 5. minidb 실행 ──
./build/minidb sql.db
```

```
minidb> .stats
minidb> .debug
minidb> SELECT * FROM users WHERE id = 500000
minidb> SELECT * FROM users WHERE email = 'user500000@test.com'
minidb> EXPLAIN SELECT * FROM users WHERE id = 500000
minidb> EXPLAIN SELECT * FROM users WHERE email = 'user500000@test.com'
minidb> .exit
```
